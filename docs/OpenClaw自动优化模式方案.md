# OpenClaw 自动优化模式方案

## 1. 目标

在现有 `Jarvis -> OpenClaw -> 人工确认` 的链路上，再增加一条“自动优化 Jarvis”支线：

1. `Jarvis` 先完成主链路匹配
2. `OpenClaw` 自动复核结果
3. `OpenClaw` 基于复核结果、人工确认结果、候选池和 trace 判断 Jarvis 的结构性问题
4. `OpenClaw` 读取指定代码模块，生成最小补丁
5. 自动跑测试和评测
6. 达到阈值后进入“待人工审核”
7. 人工决定是否合并

一句话：

不是让 OpenClaw 直接替代 Jarvis，而是让它成为 Jarvis 的“自动二审教师 + 自动补丁工”。

## 2. 适用前提

本方案适用于以下条件都成立时：

- OpenClaw 已经本地部署
- OpenClaw 已经能调用 `auto-quota` 的 `/api/openclaw/...`
- 结果页、OpenClaw 审核页和 `review-draft / review-confirm` 已经可用
- 代码仓库允许本地 agent 读取并修改受限目录

## 3. 总体架构

分成 4 个角色：

### 3.1 Jarvis

职责：

- 召回
- 候选生成
- 排序
- 最终主结果输出
- trace 输出

Jarvis 始终是主匹配引擎。

### 3.2 OpenClaw 审核器

职责：

- 审黄灯、红灯、冲突项
- 输出结构化复核结论
- 生成 `review-draft`

不直接拥有最终正式写入权。

### 3.3 OpenClaw 优化器

职责：

- 读取已复核结果和人工最终结果
- 识别 Jarvis 错误模式
- 读取指定代码
- 生成补丁
- 跑测试
- 跑评测
- 产出“是否建议合并”的结论

### 3.4 人工审核者

职责：

- 最终确认业务结果
- 最终确认代码补丁
- 决定是否合并/发布

## 4. 三种工作模式

建议把 OpenClaw 明确拆成三种模式，不要混用。

### 4.1 审单模式

只做：

- 读取任务
- 读取结果
- 读取 review-items
- 提交 `review-draft`

不读代码，不改代码。

### 4.2 诊断模式

只做：

- 读取结果
- 读取 trace
- 读取候选池
- 读取指定代码
- 分析 Jarvis 结构性问题
- 给出优化建议

不改代码。

### 4.3 优化模式

做完整闭环：

- 读取错题样本
- 读取指定代码
- 生成最小补丁
- 自动跑测试
- 自动跑评测
- 生成补丁说明

但仍然：

- 不自动部署
- 不自动推送生产
- 不自动合并主分支

## 5. 自动优化触发条件

不要每次 Jarvis 跑完都触发代码优化。

只在满足以下条件之一时触发：

1. 单任务低置信错误数超过阈值
2. `OpenClaw` 与 `Jarvis` 冲突比例过高
3. 同一类错误连续出现
4. 人工频繁把 Jarvis 改成候选池中的其它项
5. 人工搜索后选中定额，说明候选不足

建议阈值：

- 红灯项 >= 10
- `Jarvis != OpenClaw` 冲突率 >= 15%
- 某类错误连续 3 个任务出现
- 人工改写率 >= 20%

只有达到阈值，才进入自动优化模式。

## 6. OpenClaw 允许读取的数据

OpenClaw 自动优化时可读取：

- `task`
- `results`
- `review-items`
- `review-pending`
- `alternatives`
- `trace`
- `openclaw_review_payload`
- `human_feedback_payload`
- `review-confirm` 最终结果

重点不是只看“最后改成了什么”，而是看：

- Jarvis 原来为什么会错
- 候选池里有没有正确答案
- 是召回错、排序错、专业错还是参数错

## 7. OpenClaw 允许修改的代码范围

必须做“目录白名单”，不要给它全仓无限写权限。

### 7.1 第一阶段允许修改

- `src/match_pipeline.py`
- `src/match_core.py`
- `src/unified_ranking_pipeline.py`
- `src/unified_planner.py`
- `web/backend/app/services/match_service.py`
- `tests/`

### 7.2 禁止修改

- `lzc-manifest.yml`
- `docker-compose.yml`
- `scripts/`
- 数据库迁移
- `.env`
- 部署脚本
- 生产配置
- 删除历史数据的脚本

### 7.3 第二阶段可按需放开

只有第一阶段稳定后，才考虑放开：

- `src/adaptive_strategy.py`
- `src/unified_scoring_engine.py`
- 部分 `web/backend/app/api/` 读接口

## 8. 自动优化允许做的改动类型

OpenClaw 不应该一上来就改大架构。

只允许做以下 4 类最小改动：

### 8.1 召回约束修正

例如：

- 补专业限制
- 补单位限制
- 补材料/安装方式约束
- 修复错误扩召回

### 8.2 排序特征修正

例如：

- 增加大类错配惩罚
- 降低跨专业候选得分
- 增加参数匹配权重

### 8.3 候选池治理

例如：

- 候选池太宽，导致错误 top1
- 候选池缺少明显应该出现的项

### 8.4 trace 与诊断增强

例如：

- 增加错误原因埋点
- 增加 decision reason tag
- 增加 rank 前后差异记录

## 9. 明确禁止的自动改动

OpenClaw 自动优化时，禁止：

1. 同时修改 3 个以上核心模块
2. 修改数据库 schema
3. 改部署逻辑
4. 改鉴权逻辑
5. 改发布流程
6. 自动提交生产环境配置
7. 自动推送主分支
8. 自动部署线上

## 10. 自动优化工作流

建议落成下面这条固定链路：

### 阶段 A：任务完成

1. `Jarvis` 完成任务
2. 系统生成 `task_id`
3. 结果进入 `OpenClaw review-items`

### 阶段 B：自动复核

1. `OpenClaw` 自动读取 `review-items`
2. 对黄灯/红灯/冲突项提交 `review-draft`
3. 人工做 `review-confirm`

### 阶段 C：自动诊断

1. 系统统计本任务和近期任务的错误模式
2. 若达到阈值，创建 `optimization_job`
3. `OpenClaw` 读取指定代码模块
4. 输出“问题类型 -> 可能改动点”

### 阶段 D：自动补丁

1. `OpenClaw` 生成最小补丁
2. 自动补测试
3. 自动跑单元测试
4. 自动跑离线评测
5. 生成补丁报告

### 阶段 E：人工审核

1. 人工查看补丁
2. 查看测试结果
3. 查看评测结果
4. 决定是否合并

## 11. 自动补丁的验收门槛

不是能编译就算通过。

必须同时满足：

1. 目标测试全部通过
2. 不引入新的高风险回归
3. 目标错误类型有明确改善
4. 总体指标不下降

建议门槛：

- `pytest` 目标用例通过率 100%
- 针对目标错误桶的纠正率提升 >= 10%
- 绿灯正确率不下降
- 黄灯救回率提升
- 红灯误改率不上升

如果不满足，补丁自动判为“仅建议，不可合并”。

## 12. 结果产物

每次自动优化都必须产出 4 份东西：

### 12.1 结构化问题报告

内容：

- 问题类型
- 影响任务数
- 影响结果数
- 复现样本
- 可能原因

### 12.2 补丁文件

内容：

- 修改了哪些文件
- 修改了哪些函数
- 为什么改

### 12.3 测试报告

内容：

- 跑了哪些测试
- 哪些通过
- 哪些失败

### 12.4 评测报告

内容：

- 改前指标
- 改后指标
- 提升点
- 风险点

## 13. 前端需要配合的入口

前端不要只做“审核页”，还需要一个“自动优化页”。

建议在管理员侧新增一个页面：

- `OpenClaw 自动优化`

页面至少包含：

1. 待优化任务列表
2. 错误类型聚合
3. 最近自动生成的补丁
4. 测试状态
5. 评测状态
6. 人工审核入口

每条自动优化记录建议显示：

- `task_id`
- 错误桶
- 建议修改模块
- 影响样本数
- 测试是否通过
- 评测是否通过
- `查看 patch`
- `批准合并`
- `驳回`

## 14. 后端需要补的对象

建议增加一个独立对象：

- `optimization_job`

建议字段：

- `optimization_job_id`
- `source_task_id`
- `source_result_ids`
- `issue_bucket`
- `status`
- `target_files`
- `patch_summary`
- `test_summary`
- `eval_summary`
- `approved_by`
- `approved_at`
- `created_at`

建议状态：

- `pending`
- `analyzing`
- `patch_generated`
- `testing`
- `evaluating`
- `ready_for_review`
- `approved`
- `rejected`
- `aborted`

## 15. 最值得优先落地的 3 件事

不要一口气做完全部。

按优先级：

### P1

先做自动诊断，不自动改代码。

也就是：

- 聚合错误桶
- 定位问题模块
- 生成优化建议

### P2

允许生成 patch，但不自动提交。

也就是：

- 生成补丁
- 补测试
- 跑评测
- 人工审核

### P3

再做“自动创建优化任务”和“自动进入待审状态”。

## 16. 推荐落地策略

对你当前阶段，最推荐的是：

1. 先让 OpenClaw 稳定做二审
2. 再让它稳定做错误分类
3. 再让它读指定代码生成 patch
4. 最后再做自动化测试与评测联动

不要一开始就让它：

- 跑完任务
- 直接全仓改代码
- 自动提交
- 自动部署

那样风险过高，而且你也很难知道它到底改对没有。

## 17. 最终结论

如果 OpenClaw 已迁移到本地，最优方案不是只让它“继续审单”，而是升级成：

- 业务侧：`二审器`
- 工程侧：`自动诊断器`
- 研发侧：`受限补丁生成器`

但必须坚持 4 条边界：

1. Jarvis 仍然是主引擎
2. OpenClaw 只在受限目录内改代码
3. 自动改动必须带测试和评测
4. 最终合并和发布仍然由人工确认

这套方案的价值不在于“让 OpenClaw 帮你写代码”。

真正价值在于：

- 用真实错题反推 Jarvis
- 把错误分类结构化
- 让优化从“反复猜”变成“有证据地改”
