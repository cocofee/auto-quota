# 2026-03-26 安装真实清单专项总结

## 当前结论

这轮工作有意偏离了“直接把数学模型 `Constrained Gated Ranker + Accept Head` 做成主排序器”的原计划，但这是数据驱动后的主动收缩，不是跑偏。

当前更稳的落点是：

- `LTR v2` 继续做主排序
- `CGR` 只保留为严格否决和 shadow guard
- 先修入口路由、分册路由、specialty/book 种子污染
- 等候选池更干净后，再回到数学模型做 in-pool 精排

## 为什么这样定

真实安装样本上，`CGR` 直接接管主排序会明显伤分：

- `compare_ltr_v2.summary.json`: `41.3%`
- `compare_ltr_v2_cgr.summary.json`: `35.3%`

这说明当前问题不只是排序表达力不够，前面还有更重的入口污染：

- 第一跳就选错册
- seeded `specialty/book` 把候选池带歪
- 同一清单里多册混合时，分部/工作表/项目特征这些强上下文没有被充分利用

所以先把 `CGR` 收回到“强约束守门员”是对的，先避免它在脏候选池上放大错误。

## 这轮真实集结果

统一用同一份真实导入安装样本 `385` 条做对比，当前链路结果是：

- 当前可比基线 `compare_ltr_v2_shadow_baseline.summary.json`: `38.2%`，`147/385`
- 加 strict CGR shadow guard `compare_ltr_v2_shadow_guard_strict.summary.json`: `38.4%`，`148/385`
- 再加入口路由 + specialty/book 收紧 `compare_ltr_v2_shadow_guard_strict_routing_specialty_tight.summary.json`: `38.7%`，`149/385`

最终这版诊断：

- `oracle_in_candidates = 227`
- `synonym_gap = 91`
- `wrong_tier = 92`
- `wrong_book = 47`
- `no_result = 6`

结论很直接：

- 排序层还有空间，但不是当前最大瓶颈
- `wrong_book` 仍然是入口阶段问题，不先修入口，后面再强的排序器也会被脏池拖死
- 数学模型现在更适合做“候选已对册、已入池”的第二阶段

## 这轮代码动作

这一轮已经落地的改动重点有四块：

1. `src/ltr_ranker.py`
   - 明确 `LTR` 为主路径
   - 给 `CGR` 增加 shadow guard，只在 fatal hard conflict / high-conf wrong-book 等强条件下允许覆盖 top1

2. `src/context_builder.py`
   - 把 `project_name`、`bill_name` 等更强的入口上下文并入 context prior
   - 扩展系统提示词，增强给排水/电气/消防/暖通识别

3. `src/specialty_classifier.py` + `src/match_pipeline.py`
   - 加强 section / sheet / desc / feature 的分册锚点
   - 对 seeded `specialty` 的覆盖改成“只有强证据才允许推翻”
   - 增补 `C12` 绝热/刷油等强锚点，减少 `C12 -> C10` 漂移

4. 测试
   - 增加 stage-priority、CGR guard、feature context、strict book routing 覆盖

## 对数学模型的定位修正

数学模型没有被否定，只是角色变了：

- 原计划：主排序器
- 现阶段：严格 veto / shadow 分析器
- 下一阶段：在“正解已入池、分册已大体正确”的样本上重新训练精排

更准确的实施顺序应该是：

1. 先做入口和分册路由清洗
2. 再做召回层补缺
3. 最后回到 `CGR + Accept Head` 做排序和拒识

## 下一步

下一阶段优先做：

1. 专攻 seeded `specialty/book` 污染，继续压 `wrong_book`
2. 收敛 `C8 ↔ C10`、`C4 ↔ C5`、`C12 → C10` 这些高频错册簇
3. 在真实集上继续统一口径评测，禁止再拿不同样本集制造“假提升”
4. 等入口稳定后，再单独切一批 `oracle_in_candidates = 1` 的样本回训数学模型
