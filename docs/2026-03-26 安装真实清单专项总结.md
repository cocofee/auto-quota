# 2026-03-26 安装真实清单专项总结

## 当前结论

这轮工作已经把重点从“直接让 `Constrained Gated Ranker + Accept Head` 接管主排序”收回到“先把入口和评测口径做干净”。

当前稳定结论是：

- 主排序仍以 `LTR v2` 为准
- `CGR` 暂时只适合做强约束守门和 shadow 分析
- 入口侧要优先修 `section/sheet/item` 这些本地强证据
- `project_name/file_name` 只保留为元数据和软上下文，不进入主路由裁决

## 为什么这样定

真实安装样本表明，当前主要问题不是“排序公式不够复杂”，而是“第一跳就容易进错册，后面再强的排序也会被脏候选池拖死”。

已确认的风险点：

- `wrong_book` 仍然是入口阶段问题
- 同一份清单混多册时，真正有效的是 `section/sheet/相邻条目/项目特征`
- 文件名和项目名可以提供系统感知，但不适合作为主路由硬锚点

这意味着数学模型没有被否定，只是实施顺序要后移：先把候选池洗干净，再回到 in-pool 精排。

## 评测口径修正

这次顺手修掉了一个关键偏差：之前离线真实集评测没有完整透传 `project_name` 等入口上下文，导致历史分数和实际运行口径不完全一致。

本轮已统一：

- 导入侧补齐 `source_file_name/source_file_stem/source_file_title`
- `context_prior` 能回填 `project_name/bill_name`
- `run_real_eval.py` 与 `export_real_eval_set.py` 透传同样字段

统一口径后，当前可对比基线是：

- `compare_ltr_v2_eval_aligned_no_title_anchor.summary.json`: `37.7%`，`145/385`

这比之前常看的 `39.0%` 更可信，因为它和真实链路上下文一致。

## 这轮实验结论

围绕“文件名/项目名能不能直接做主路由锚点”做了几组实验，结果都不理想：

- `compare_ltr_v2_file_entry_anchor_eval_aligned.summary.json`: `37.7%`
- `compare_ltr_v2_file_entry_anchor_soft.summary.json`: `38.2%`
- `compare_ltr_v2_file_entry_anchor_weak_only.summary.json`: `37.7%`

结论很明确：

- 文件标题信息有价值，但更适合做 `metadata + soft context`
- 一旦把它抬成主路由锚点，会增加 `wrong_book`
- 所以不能让 `project_name/bill_name/source_file_*` 直接决定 book routing

## 对数学模型的定位修正

数学模型现在的正确位置是：

- 不是第一优先级主排序器
- 而是“候选已基本对册之后”的第二阶段精排器

更合理的顺序：

1. 清洗入口路由和分册路由
2. 修召回层缺口
3. 最后回到 `CGR + Accept Head`

否则就是在错误候选池上调复杂模型，收益会非常不稳定。

## 下一步

下一步应继续抓三件事：

1. 只用 `section/sheet/item/neighbor` 这类本地强证据压 `wrong_book`
2. 保持真实导入安装集 `385` 条作为统一评测集，禁止再混口径看“假提升”
3. 等入口稳定后，再单独切 `oracle in candidates` 样本回训数学模型
