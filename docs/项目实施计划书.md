## 一、项目概述

### 1.1 项目目标

构建一套基于 **混合搜索(向量+BM25) + 大模型精选 + 经验库学习** 的自动套定额系统，匹配准确率目标 ≥85%，支持多专业、多省份，输出可直接导入广联达。

### 1.2 技术路线佐证

| 佐证来源 | 内容 |
|----------|------|
| OpenClaw（GitHub 14.5万星） | 验证了向量70%+BM25 30%混合搜索的有效性 |
| BIM+LLM论文（MDPI 2026.01） | 验证了大模型在工程造价领域的可行性 |
| NLP集成方法论文 | 验证了多种搜索算法融合的匹配架构 |
| 行业商业产品（智多星、广联达AecGPT等） | 验证了市场需求真实存在 |

### 1.3 竞品对比

| 特性 | 同类工具 | 智多星 | 我们的系统 |
|------|--------|--------|-----------|
| 准确率 | 50-60% | 未知 | 目标85%+ |
| 学习能力 | 无 | 未知 | 经验库+用户修正 |
| 编码依赖 | 依赖 | 未知 | 不依赖（语义匹配） |
| 多省支持 | 有限 | 有限 | 一套引擎+多省数据包 |
| 数据控制 | 云端 | 云端 | 本地运行 |
| 成本 | 收费 | 收费 | API费用（几分钱/条） |
| 可定制 | 不可 | 不可 | 完全可定制 |

## 二、技术选型与理由

### 2.1 技术栈

| 组件 | 选型 | 选择理由 |
|------|------|----------|
| **语言** | Python 3.12 | 用户已安装；AI/ML生态最成熟；Streamlit原生支持 |
| **向量模型** | BGE-large-zh-v1.5 | 中文向量模型MTEB榜单前列；本地GPU运行无API费用；~2GB显存 |
| **向量数据库** | ChromaDB | 纯Python、安装简单；几万条数据性能足够；无需单独部署服务 |
| **关键词搜索** | rank_bm25 | 纯Python实现；与向量搜索互补，精确关键词召回 |
| **中文分词** | jieba | 支持自定义词典；可添加工程造价专业术语 |
| **大模型API** | Claude/GPT/DeepSeek | 多模型切换降低依赖风险；DeepSeek性价比高；Claude/GPT准确度高 |
| **数据库** | SQLite | 零配置、文件级；Python内置支持；单用户场景完全够用 |
| **Excel读写** | openpyxl | 支持xlsx格式读写；保持格式和样式；有XML直接解析备选方案 |
| **UI框架** | Streamlit | 纯Python（不需要前端知识）；`st.data_editor`支持可编辑表格；原生Excel上传/下载；界面美观现代 |
| **硬件** | RTX 4070 12GB | 用户已有；BGE模型只需~2GB显存；富余空间大 |

### 2.2 不选的方案及原因

| 被排除方案 | 排除原因 |
|-----------|----------|
| FAISS向量库 | 需要手动管理索引，ChromaDB更简单 |
| Elasticsearch | 需要Java+独立部署，对单用户场景过重 |
| Gradio UI | 不支持可编辑表格，界面不如Streamlit美观 |
| Electron+React | 需要前端开发知识，开发量大，维护复杂 |
| MySQL/PostgreSQL | 需要安装数据库服务，SQLite足够 |
| 微调本地模型 | 数据量不够，成本高，通用大模型+RAG更实际 |

#### 多Agent动态路由（渐进式纠偏架构）

核心逻辑：**系统越不成熟 → 多Agent越活跃 → 系统越成熟 → 多Agent越少介入**

```
输入清单
  │
  ▼
【第1层：经验库直通】
  ├─ 命中且置信度>90% → 直接返回（0费用，毫秒级）
  │
  └─ 未命中或置信度不够 ↓

【第2层：混合搜索 + 单Agent匹配】
  │  搜索召回Top 20 → 大模型精选（DeepSeek/Claude）
  │
  ├─ 置信度>85% → 仅做参数快速验证 → 输出（低费用）
  │
  └─ 置信度<85% ↓

【第3层：多Agent纠偏（仅低置信度触发）】
  │
  ├──→ 参数审核Agent："DN对不对？截面档位对不对？"
  ├──→ 规则审核Agent："要不要加试压？冲洗？调试？"
  │
  ▼
  裁判Agent：综合意见 → 修正匹配结果 → 输出
```

**系统进化后的费用变化**：
```
初期（经验库空）：
  80%走第3层（多Agent）→ API费用高 ≈ ¥0.15/条

中期（几百条经验）：
  80%走第1层（直通）→ 费用 ≈ ¥0.02/条

后期（经验丰富）：
  95%走第1层 → 费用 ≈ ¥0.005/条，近乎免费
```

## 七、从OpenClaw借鉴的设计模式

### 7.1 已纳入第一阶段的设计

| 模式 | 来源 | 在我们系统中的应用 |
|------|------|-------------------|
| **混合检索70/30配比** | OpenClaw Memory System | hybrid_searcher.py中向量权重0.7、BM25权重0.3 |
| **模型分层路由** | OpenClaw Model Routing | llm_matcher.py中根据复杂度选择DeepSeek或Claude |
| **持久化优先** | OpenClaw Memory Flush | 匹配结果和用户修正立即写入SQLite+ChromaDB |
| **优雅降级** | OpenClaw Fallback Chain | GPU异常→BM25，API异常→纯搜索候选 |

### 7.2 纳入未来阶段的设计

| 模式 | 来源 | 计划应用 |
|------|------|----------|
| **AgentSkills标准** | OpenClaw Skills | 匹配引擎封装为SKILL.md格式，接入OpenClaw生态 |
| **技能进化** | OpenClaw Recursive Skills | 匹配失败→修正→提取规则→自动封装为新的匹配模式 |
| **多通道接入** | OpenClaw Gateway | 通过微信/钉钉查询定额（第三阶段） |
| **会话隔离** | OpenClaw Session | 不同项目的匹配上下文互相隔离 |

# 自动套定额技能

当用户描述一个工程清单项时，自动从定额库中匹配最合适的定额子目。

## 使用方法
用户发送类似 "DN150镀锌钢管沟槽连接该套什么定额" 的消息时，
运行 `scripts/match.py` 进行匹配并返回结果。

## 输出格式
返回匹配的定额编号、名称、工程量，以及置信度评分。

## 九、质量保证

### 9.1 代码质量

- 每个文件单一职责，不超过500行
- 关键函数必须有中文文档字符串
- 错误处理使用try/except，提示信息用中文
- 使用loguru记录关键操作日志

### 9.2 测试策略

| 测试层级 | 方法 | 关注点 |
|----------|------|--------|
| 单元测试 | 每个模块独立测试 | text_parser参数提取准确性 |
| 集成测试 | 搜索→匹配→输出链路测试 | 数据在模块间正确流转 |
| 端到端测试 | 用真实清单走完全流程 | 输出Excel可导入广联达 |
| 对比测试 | 与同类工具结果对比 | 我们的准确率 ≥ 同类工具 |

### 9.3 对比测试方案

1. 取同类工具的测试项目消防清单作为输入
2. 分别用同类工具和我们的系统匹配
3. 人工判定每条匹配是否正确
4. 统计并对比准确率
