> 文档版本：v1.0
> 创建日期：2026-02-13
> 项目代号：auto-quota

## 一、项目背景

### 1.1 行业现状

工程造价中，"套定额"是将工程量清单项与定额子目进行匹配的核心工作。目前行业内的做法：

- **纯手工**：造价人员逐条在定额库中查找匹配，效率低、耗时长
- **小栗AI**：市面上的AI辅助工具，成功率约50-60%，大量需要手动修正
- **广联达AecGPT**：集成在广联达软件中，但封闭且收费
- **智多星造价AI**：商业产品，号称1000条5分钟

### 1.2 痛点分析

| 痛点 | 说明 |
|------|------|
| 匹配不准 | 现有AI工具成功率低，错误的匹配比没有匹配更麻烦 |
| 编码不靠谱 | 清单编码经常有误（流水号、编错专业），导致连锁错误 |
| 修正成本高 | 改清单→再套定额→再改定额，反复折腾 |
| 不会学习 | 同样的错误反复出现，工具不会从修正中学习 |
| 关联定额遗漏 | 管道需要加试压、冲洗等附加定额，工具经常遗漏 |
| 跨专业困难 | 一个项目可能需要安装+土建+市政多专业定额 |

### 1.3 项目目标

建设一套自动套定额系统，实现：

1. **匹配准确率 ≥ 85%**（高于小栗AI的50-60%）
2. **越用越准**：通过经验库和用户修正持续提升
3. **不依赖编码**：纯语义匹配，不受编码质量影响
4. **多专业支持**：安装、土建、市政全覆盖
5. **多省份扩展**：一套引擎，多省数据包
6. **输出可用**：直接导入广联达，减少人工操作

## 二、目标用户

### 2.1 用户画像

| 属性 | 描述 |
|------|------|
| 角色 | 工程造价人员 |
| 技术水平 | 非程序员，但熟练使用Excel和广联达 |
| 工作场景 | 投标报价、结算审核、概预算编制 |
| 使用频率 | 每个项目使用1-3次（编清单+套定额） |
| 期望 | 减少重复劳动，提高准确率，把时间花在审核而不是查找上 |

### 2.2 使用场景

**场景1：标准流程**
> 用户拿到工程量清单Excel → 上传到系统 → 系统自动匹配定额 → 用户在可编辑表格中审核/修正 → 下载广联达可导入的Excel

**场景2：从GQI工程量报表出发（未来）**
> 用户导出GQI工程量报表 → 系统自动编清单 → 自动套定额 → 审核/修正 → 导入广联达

**场景3：经验积累**
> 用户审核修正匹配结果 → 系统自动学习 → 下次遇到类似清单自动使用正确匹配

**场景4：快速查询（未来，OpenClaw集成）**
> 用户通过微信/钉钉发送"DN150镀锌钢管沟槽连接该套什么定额" → AI助手直接回答

## 三、功能需求

### 3.1 第一阶段 - 核心功能（MVP）

#### F1: 定额数据库管理
- **F1.1** 导入定额Excel文件（支持安装/土建/市政多专业）
- **F1.2** 自动解析定额：编号、名称、特征参数、单位、工作类型
- **F1.3** 结构化提取参数：管径(DN)、截面(mm²)、电流(A)、重量(t)等
- **F1.4** 向量化存储：BGE模型生成向量，存入ChromaDB
- **F1.5** BM25索引：jieba分词 + 工程造价专业词典
- **F1.6** 定额搜索：支持关键词搜索和语义搜索

#### F2: 清单读取
- **F2.1** 支持标准12位编码格式的清单Excel
- **F2.2** 支持流水号格式（非标准编码）
- **F2.3** 支持小栗AI输出格式
- **F2.4** 自动识别列名映射（序号、编码、名称、特征、单位、工程量）
- **F2.5** 结构化提取清单特征参数

#### F3: 混合搜索匹配
- **F3.1** BM25关键词搜索 → 召回Top 20候选
- **F3.2** 向量语义搜索 → 召回Top 20候选
- **F3.3** RRF（Reciprocal Rank Fusion）融合排序
- **F3.4** 默认配比：向量70% + BM25 30%（参考OpenClaw经验值）

#### F4: 大模型精选
- **F4.1** 支持多种大模型API：Claude、GPT、DeepSeek（可切换）
- **F4.2** 模型分层路由（借鉴OpenClaw）：
  - 经验库命中 → 零成本直接返回
  - 简单匹配 → 便宜模型（DeepSeek）
  - 复杂/低置信 → 高端模型（Claude）
- **F4.3** 结构化Prompt，包含：清单描述 + 候选列表 + 参考案例（few-shot）
- **F4.4** 返回JSON格式：主定额 + 关联定额 + 置信度 + 工程量换算
- **F4.5** 支持并发调用（5路并发加速）

#### F5: 参数验证
- **F5.1** 数值参数精确匹配：DN档位、截面档位、电流档位
- **F5.2** 单位兼容性检查
- **F5.3** 参数不匹配时降低置信度

#### F6: 多智能体动态纠偏（渐进式架构）

核心逻辑：**系统越不成熟 → 多Agent越活跃 → 系统越成熟 → 多Agent越少介入**

- **F6.1** 动态路由器：根据经验库覆盖度和匹配置信度，自动决定走几层Agent
- **F6.2** 四层渐进架构：
  - **第1层-经验库直通**：命中且置信度>90% → 直接返回（0费用、毫秒级）
  - **第2层-单Agent匹配**：搜索+单个大模型精选（低费用、3-6秒）
  - **第3层-参数快速验证**：单Agent结果置信度>85%时，仅做参数校验（低费用）
  - **第4层-多Agent纠偏**：置信度<85%时，启动完整多Agent协作：
    - 参数审核Agent：检查DN/截面/电流等数值参数
    - 规则审核Agent：检查定额说明规则、附加工序是否遗漏
    - 裁判Agent：综合审核意见，给出最终匹配+理由
- **F6.3** 系统进化效果：
  - 初期（经验库空）：80%清单走第4层 → API费用高，但准确率有保障
  - 中期（几百条经验）：80%走第1层，20%走第3-4层 → 费用大幅降低
  - 后期（经验丰富）：95%+走第1层 → 近乎免费，极少数新类型才触发多Agent

#### F7: 置信度评估与标记
- **F7.1** 绿色（>85%）：自动确认
- **F7.2** 黄色（60-85%）：建议人工确认
- **F7.3** 红色（<60%）：需人工处理

#### F7: 结果输出
- **F7.1** 生成广联达可导入格式的Excel（小栗AI格式）
- **F7.2** 清单行 + 定额行交替排列
- **F7.3** 包含置信度颜色标记
- **F7.4** 工程量自动单位换算（m→10m、个→10个等）

#### F8: Streamlit Web界面
- **F8.1** 主页：系统概览、使用指南
- **F8.2** 匹配定额页：
  - 拖拽上传清单Excel
  - 实时匹配进度条
  - 可编辑表格展示匹配结果（颜色标记）
  - 用户可在表格中修改匹配
  - 一键下载结果Excel
- **F8.3** 定额数据库页：查看/导入/搜索定额
- **F8.4** 经验库页：查看历史匹配记录、统计成功率
- **F8.5** 设置页：API密钥、省份选择、参数调整

### 3.2 第二阶段 - 增强功能

#### F9: 经验库与学习机制
- **F9.1** 从已完成项目批量导入历史匹配数据
- **F9.2** 用户每次修正自动存入经验库
- **F9.3** 新清单优先在经验库中搜索相似项
- **F9.4** 统计各类别匹配成功率
- **F9.5** 自动调整置信度阈值
- **F9.6** 持久化优先原则（借鉴OpenClaw）：匹配结果立即写盘

#### F10: 定额规则知识库
- **F10.1** 导入各省定额章节说明、计算规则
- **F10.2** 知识文字向量化，存入ChromaDB独立collection
- **F10.3** 匹配时自动检索相关规则注入Prompt
- **F10.4** 支持多省份规则并存

#### F11: 综合知识库
- **F11.1** 支持导入群聊经验记录
- **F11.2** 支持导入各省定额补充说明
- **F11.3** 支持导入各类文件（txt/pdf/截图文字）
- **F11.4** 自动解析、分段、向量化
- **F11.5** 知识优先级：经验库 > 规则库 > 经验文库 > 参考算法

#### F12: 多专业扩展
- **F12.1** 导入土建定额
- **F12.2** 导入市政定额
- **F12.3** 跨专业搜索和匹配
- **F12.4** 按项目需要自动选择专业范围

#### F13: 多省份架构
- **F13.1** 一套引擎 + 多省数据包设计
- **F13.2** 通用层（匹配引擎、经验知识）全国共享
- **F13.3** 专属层（定额数据、规则说明）各省独立
- **F13.4** 一键导入新省份定额

#### F14: 优雅降级（借鉴OpenClaw）
- **F14.1** GPU不可用 → 降级到纯BM25搜索
- **F14.2** API不通 → 返回搜索候选让用户手选
- **F14.3** 向量库异常 → 回退到关键词匹配
- **F14.4** 所有降级过程对用户透明提示

### 3.3 第三阶段 - 完整产品

#### F15: GQI工程量报表 → 自动编清单
- **F15.1** 读取GQI导出的工程量报表
- **F15.2** 自动生成标准工程量清单
- **F15.3** 然后自动套定额（串联F3-F7）

#### F16: 价格数据库
- **F16.1** 每个项目完成后自动保存综合单价和主材价格
- **F16.2** 新项目自动对比历史价格，标记异常偏高/偏低
- **F16.3** 主材价格趋势追踪
- **F16.4** 快速估价功能
- **F16.5** 市场报价参考

#### F17: OpenClaw集成（未来方向）
- **F17.1** 将匹配引擎封装为AgentSkills标准格式
- **F17.2** 支持通过微信/钉钉查询定额匹配
- **F17.3** 支持自然语言对话式查询
- **F17.4** 技能进化机制：匹配失败案例 → 修正 → 自动提取规则

## 四、非功能需求

### 4.1 性能要求

| 指标 | 要求 |
|------|------|
| 单条匹配耗时 | ≤ 6秒（含API调用） |
| 100条清单批量处理 | ≤ 3分钟（5路并发） |
| 经验库命中返回 | ≤ 100毫秒 |
| 向量搜索响应 | ≤ 100毫秒 |
| BM25搜索响应 | ≤ 50毫秒 |
| UI页面加载 | ≤ 3秒 |
| Excel导出 | ≤ 10秒（1000行以内） |

### 4.2 准确率要求

| 阶段 | 目标准确率 | 说明 |
|------|-----------|------|
| 初始（无经验库） | ≥ 70% | 仅靠混合搜索+大模型 |
| 导入经验库后 | ≥ 85% | 经验库覆盖常见清单 |
| 持续使用后 | ≥ 90% | 经验库持续积累 |

### 4.3 安全要求

- API密钥存储在`.env`文件，不进入版本控制
- 定额数据和经验库存储在本地，不上传云端
- 向量搜索在本地GPU运行，数据不外泄
- 仅大模型API调用涉及网络传输（清单描述+候选列表）

### 4.4 兼容性要求

| 项目 | 要求 |
|------|------|
| 操作系统 | Windows 10/11 |
| Python版本 | 3.12+ |
| GPU | NVIDIA RTX 4070 12GB（或同等以上） |
| 浏览器 | Chrome/Edge（Streamlit UI） |
| 广联达版本 | 支持导入小栗AI格式的Excel |

### 4.5 可维护性要求

- 代码注释使用中文
- 每个文件单一职责（一个文件做一件事）
- 错误提示使用中文，说明问题原因和解决方法
- 配置项集中管理（config.py + .env）
- 日志记录关键操作（匹配过程、API调用、错误信息）

## 五、数据需求

### 5.1 定额数据

| 数据项 | 来源 | 格式 |
|--------|------|------|
| 安装工程定额 | 已有（`C 通用安装工程_全部.xlsx`） | Excel，103个Sheet |
| 土建工程定额 | 用户从广联达导出 | Excel，格式同上 |
| 市政工程定额 | 用户从广联达导出 | Excel，格式同上 |

每条定额包含4列：
- A列：定额编号（如 C1-1-1）
- B列：名称+特征参数（如"水灭火系统 水喷淋钢管 公称直径150"）
- C列：计量单位（如 台、m、10m）
- D列：工作类型（如 安装）

### 5.2 清单数据（输入）

支持的格式：
1. **标准格式**：12位编码 + 项目名称 + 项目特征描述 + 计量单位 + 工程量
2. **流水号格式**：序号(1,2,3...) + 项目名称 + 项目特征描述 + 计量单位 + 工程量
3. **小栗AI格式**：清单行+定额行交替排列

### 5.3 输出数据

广联达可导入格式Excel，参照小栗AI输出：
- 清单行：序号 | 12位编码 | 项目名称 | 项目特征描述 | 计量单位 | 工程量
- 定额行：（空） | 定额编号 | 定额名称+参数 | （空） | 计量单位 | 定额工程量

### 5.4 经验库数据

| 字段 | 说明 |
|------|------|
| bill_name | 清单项目名称 |
| bill_description | 项目特征描述 |
| bill_unit | 计量单位 |
| quota_ids | 匹配的定额编号列表 |
| quota_names | 匹配的定额名称列表 |
| source | 数据来源（项目导入/用户修正） |
| province | 定额所属省份 |
| confidence | 置信度 |
| create_time | 创建时间 |
| update_time | 更新时间 |

### 7.2 借鉴OpenClaw的设计模式

| 设计模式 | OpenClaw原版 | 我们的应用 |
|----------|-------------|-----------|
| 混合检索配比 | 向量70%+BM25 30% | 定额搜索使用相同配比 |
| 模型分层路由 | 简单→小模型，复杂→大模型 | 经验命中→免费，简单→DeepSeek，复杂→Claude |
| 持久化优先 | 先写磁盘再遗忘 | 匹配结果立即写入经验库 |
| 优雅降级 | 嵌入失败→纯BM25 | GPU异常→BM25，API异常→手动选择 |
| 技能进化 | 失败→写代码→封装Skill | 失败→修正→提取规则→存入经验库 |
| AgentSkills标准 | 模块化技能插件 | 引擎按AgentSkills格式封装，未来接入OpenClaw |

## 九、风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 大模型API不稳定 | 中 | 匹配中断 | 优雅降级为纯搜索候选 |
| 定额Excel格式不标准 | 高 | 导入失败 | 兼容多种格式，ZIP直接解析XML |
| BGE模型显存不足 | 低 | 向量搜索不可用 | 降级到BM25搜索 |
| 匹配准确率不达标 | 中 | 用户体验差 | 调整搜索配比、优化Prompt、积累经验库 |
| 广联达格式变化 | 低 | 导出不可用 | 输出模块可配置，适应格式变化 |
