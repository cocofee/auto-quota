# Jarvis 快速提效实施方案

## 1. 目标

这份方案不追求一次性重写算法，而是用最短路径把 Jarvis 的正确率、可控性、迭代速度同时拉起来。

核心原则只有 3 条：

1. 先把低质量样本分流，减少 Jarvis 在明显脏样本上的误判。
2. 让 OpenClaw 只做二审和失败归因，不直接替代 Jarvis 主链路。
3. 让人工审核结果稳定回流，形成可持续学习闭环。

## 2. 当前代码里已经有的积木

现有代码并不是能力不够，而是没有完全串起来：

- 主匹配链路：`src/match_pipeline.py`
- 统一排序骨架：`src/unified_ranking_pipeline.py`
- 反馈学习：`src/feedback_learner.py`
- 运行和知识命中统计：`src/accuracy_tracker.py`
- OpenClaw 结构化二审上下文：`web/backend/app/services/openclaw_review_service.py`

这说明现在最应该做的不是继续堆新规则，而是把检索、排序、二审、人工修正、回流这条闭环真正打通。

## 3. 现阶段真正的瓶颈

### 3.1 检索和排序还没有彻底统一

仓库里已经有统一排序骨架，但 `src/unified_ranking_pipeline.py` 还处在 skeleton 状态，没有完全变成生产主通路。

结果是：

- 一部分逻辑在旧链路里硬编码
- 一部分逻辑在显式 picker 里单点修
- 一部分逻辑在 review 阶段兜底

这会导致每次修一个错例，都像在补洞。

### 3.2 低质量输入没有被前置分流

很多错配并不是 Jarvis 不会，而是清单文本太短、太脏、缺字段、描述混杂。

如果这种样本直接进入主排序，后面再怎么调分都容易反复横跳。

### 3.3 OpenClaw 现在更像展示层，不像真正二审层

现在已经有 OpenClaw 审核上下文和桥接接口，但它更多是在看 Jarvis 结果，还没有稳定承担：

- 红黄灯项目二审
- Top 候选重排
- 失败归因聚类
- 审核意见回流

### 3.4 人工审核结果没有成为高质量训练样本

`src/feedback_learner.py` 已经支持从修正结果写经验库，但实际业务上还缺少两层约束：

- 只把高质量人工确认结果沉淀为强样本
- 把错因类别一起保存，而不是只存正确答案

否则经验库只会越来越大，不会越来越聪明。

## 4. 最合理也最快的提效路线

建议分 3 个阶段，每个阶段都能独立验收。

## 第一阶段：先把错误分流和二审跑顺

这是最快能看到效果的一阶段，优先级最高。

### 4.1 Jarvis 主链路只做两件事

1. 给出 Top1 和 TopN 候选
2. 给出是否值得二审的明确标记

不要要求 Jarvis 一次就完美命中所有难项。

Jarvis 先把任务做成：

- 绿灯：直接通过
- 黄灯：进入 OpenClaw 二审
- 红灯：直接进人工审核

重点不是把所有题一次做对，而是把错题稳定送到正确的后续处理层。

### 4.2 OpenClaw 只审该审的

OpenClaw 不要全量审核，只审核：

- 红灯
- 黄灯
- Top1 与 Top2 分差很小
- 命中了明显冲突规则
- 文本过短、脏、歧义高

这样会立刻减少二审成本，也能让 OpenClaw 的建议质量稳定下来。

### 4.3 OpenClaw 审核权限边界

建议权限设计如下：

- 可以在 Jarvis 给出的候选池里改排位
- 可以建议维持 Jarvis
- 可以建议人工处理
- 可以触发重新检索一次
- 不能直接无痕写正式结果

正式落库仍然要人工确认，这和当前业务要求一致。

### 4.4 第一阶段验收指标

第一阶段不看总准确率先提升多少，先看这 4 个指标：

- 黄红灯占比是否下降
- OpenClaw 二审采纳率是否高于 60%
- 人工平均处理时长是否下降
- Top1 错大类问题是否明显下降

## 第二阶段：把反馈闭环做实

第一阶段稳定后，立刻做反馈闭环，否则前面的提升会停住。

### 5.1 人工审核结果分 3 类回流

每条人工结果不要只记录最后选了哪个定额，还要记录：

1. 维持 Jarvis
2. 采纳 OpenClaw
3. 人工另选

这三类样本价值完全不同。

其中最关键的是人工另选，因为这是最强的负样本和正样本对。

### 5.2 回流时必须保存错因标签

建议至少记录下面这些错因：

- 错专业
- 错大类
- 错对象
- 参数档位错
- 安装方式错
- 材质错
- 候选不足
- 搜索词不对

这样后续才能知道应该修：

- query builder
- retrieval
- rerank
- review rules
- 经验库

否则你只知道错了，不知道该改哪层。

### 5.3 经验库只吃强确认样本

建议经验回流分层：

- authority：人工明确确认且可复用的样本
- candidate：OpenClaw 建议但还未形成长期共识的样本
- reject_log：被人工否掉的候选和错因

不要把所有修正都直接喂进 authority 层。

### 5.4 第二阶段验收指标

- 同类错因是否连续 3 天下降
- 经验库直通命中率是否上升
- 人工另选样本中，重复错因占比是否下降
- OpenClaw 被人工否决的原因是否可归类

## 第三阶段：让 OpenClaw 参与离线优化，不要直接在线乱改

你现在把 OpenClaw 部署到本地，这是优势，但要用对位置。

最好的方式不是让它在线直接替 Jarvis 决策，而是让它承担离线优化任务。

### 6.1 OpenClaw 最适合做的 4 件事

1. 对审核失败样本做错因聚类
2. 生成 query、rerank、review rule 的改进建议
3. 基于固定样本集跑 A/B 对比
4. 在受控目录里提交小步代码修改建议

也就是：

- 在线上，它是二审员
- 在线下，它是优化助手

### 6.2 建议的自动优化模式

每日或每批任务结束后，自动生成一个失败样本包给 OpenClaw，内容包括：

- 原始清单
- Jarvis Top1 和 Top3
- OpenClaw 建议
- 人工最终结果
- 错因标签
- trace 摘要

然后让 OpenClaw 只做 3 类输出：

1. 规则修补建议
2. 检索词修补建议
3. 候选重排逻辑建议

不要允许它直接改数据库里的正式业务结果。

### 6.3 本地 OpenClaw 直接读代码的正确用法

可以让它直接读代码并提 patch，但必须有约束：

- 只允许改白名单目录
- 每次只改一个问题域
- 必须附带对应测试
- 必须跑固定 benchmark
- 人工看 diff 后再合入

推荐优先开放的目录：

- `src/query_builder.py`
- `src/query_router.py`
- `src/match_pipeline.py`
- `src/review_checkers.py`
- `src/review_correctors.py`
- `tests/`

不建议一开始就让它跨很多模块一起大改。

## 5. 具体技术路线建议

### 7.1 先统一候选池

你现在最需要统一的不是 UI，而是候选池。

Jarvis、OpenClaw、人工三层都应该围绕同一份候选池工作：

- Jarvis 定额：当前 Top1
- OpenClaw 审核：对候选池的二次判断
- 人工审核：从同一候选池里直接选，或者补搜

这样三层说的是同一个东西，后续反馈才不会乱。

### 7.2 让前端承担快速决策，不要承担推理

前端页面只做这些事：

- 一行内看清 Jarvis、OpenClaw、人工 三个定额
- 候选里直接点选
- 支持补搜正确定额
- 支持一键标记都错了
- 支持按错因批量筛选

不要把 explanation、trace、候选细节默认展开。

### 7.3 把统一排序骨架真正接成主链路

`src/unified_ranking_pipeline.py` 既然已经有，就不要长期停留在骨架状态。

建议后续逐步把这些能力往统一排序里收：

- retrieval sources
- unified score
- hard/soft constraint
- confidence calibration
- diagnostics

否则会一直在旧链路和新链路之间双轨维护。

### 7.4 JioNLP 可以用，但只能做辅助层

如果目标是前端处理更好、搜索更准，JioNLP 可以考虑放在文本预处理层，适合做：

- 文本清洗
- 词语切分辅助
- 数值和实体抽取补充
- 噪声文本归一化

但它不应该被当成核心提效手段。

真正决定正确率的仍然是：

- 候选召回是否对
- 排序是否稳定
- 二审是否有效
- 反馈是否回流

JioNLP 只能提升前置特征质量，不能代替业务规则和排序逻辑。

## 6. 最推荐的执行顺序

严格按下面顺序做，不要再同时改很多层：

1. 先完成 Jarvis 绿黄红分流
2. 再让 OpenClaw 只审核黄红灯
3. 再把人工结果按错因标签回流
4. 再扩大经验库 authority 和 candidate 分层
5. 最后才让 OpenClaw 参与离线代码优化

如果顺序反过来，比如先让 OpenClaw 大量自动改代码，结果大概率还是会乱。

## 7. 每一步怎么验收

### 第一步验收

- 红黄灯是否被稳定筛出
- OpenClaw 是否只处理该处理的项目

### 第二步验收

- 人工审核页是否能在一行内完成决定
- 是否能直接从候选里选或搜索

### 第三步验收

- 审核结果是否能稳定回写
- 再次进入任务时审核状态是否不丢

### 第四步验收

- 反馈样本是否进入分层知识库
- 同类错题是否减少

### 第五步验收

- OpenClaw 生成的 patch 是否附带测试
- benchmark 是否优于基线

## 8. 一句话结论

Jarvis 想快速提高，最合理的路不是继续单点调规则，而是：

先把 Jarvis 做成稳定初审机，再让 OpenClaw 做受控二审，再把人工结果带错因回流，最后让本地 OpenClaw 只做离线优化和小步改码。

这样提效最快，也最不容易越改越乱。
