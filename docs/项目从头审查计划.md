# 项目从头审查计划（给 Claude 执行版）

更新时间：2026-02-22  
适用项目：`auto-quota`

## 1. 审查目标

- 从“可运行”提升到“可验证、可维护、可持续演进”。
- 找出会导致错误定额、性能退化、数据污染、回归失效的关键风险。
- 形成可落地的修复批次（P0/P1/P2）和门禁策略。

## 2. 审查范围

- 核心链路：`main.py`、`src/match_engine.py`、`src/match_pipeline.py`、`src/match_core.py`
- 检索与规则：`src/query_builder.py`、`src/rule_validator.py`、`src/review_checkers.py`
- 数据层：`src/quota_db.py`、`src/experience_db.py`
- 流水线与工具：`tools/jarvis_pipeline.py`、`tools/jarvis_auto_review.py`、`tools/system_health_check.py`
- 门禁脚本：`scripts/system_health.bat`、`scripts/ai_pair_guard.bat`

## 3. 启动基线（已执行）

- 已执行：`python tools/system_health_check.py --mode full`
- 结果：通过（Required failures = 0，Optional failures = 0）
- 报告：
  - `output/health_reports/health_full_20260222_214242.json`
  - `output/health_reports/health_full_20260222_214242.md`

## 4. 分阶段审查计划

### 阶段A：架构与依赖体检（1天）

检查项：

- 梳理调用链（CLI -> 编排 -> 匹配 -> 审核 -> 经验回灌）
- 检查循环依赖、跨层直接调用、重复逻辑分叉
- 标注“高耦合入口”和“不可替换点”

产出：

- 架构风险清单（含文件与行号）
- 重构优先级（先包裹再替换）

### 阶段B：正确性审查（2天）

检查项：

- 经验库命中校验是否存在“绕过硬校验”路径
- query 规范化是否存在强规则误归类
- 候选构建/兜底替换是否会吞掉异常并误判成功
- 规则族缺失、提参失败、空值字段的兜底路径

产出：

- P0/P1 逻辑缺陷清单
- 每个缺陷的最小复现样例
- 对应回归测试草案

### 阶段C：数据与一致性审查（1天）

检查项：

- `quota_db` / `experience_db` schema 与迁移策略
- 导入历史去重、哈希、状态字段一致性
- 候选层/权威层晋升流程是否可追溯

产出：

- 数据一致性风险清单
- 迁移与回滚建议

### 阶段D：性能与稳定性审查（1天）

检查项：

- 搜索、重排、校验、Agent 调用耗时分布
- 快通道命中率、抽检不一致率
- 批处理脚本在离线/网络异常下的退化行为

产出：

- 性能瓶颈列表（按收益排序）
- 快速优化项（无需改架构）

### 阶段E：测试与门禁补强（1天）

检查项：

- 当前测试是否覆盖关键分支（经验直通、规则兜底、异常输入）
- 体检项是否覆盖新增模块
- 是否可形成“改动即回归”的最小门禁

产出：

- 必补测试清单
- 门禁命令统一入口

## 5. 交付物模板（Claude 每阶段都要交）

- `findings.md`：按严重级别列问题（必须含文件行号）
- `patch_summary.md`：改动摘要、影响面、回滚方式
- `validation.md`：执行了哪些命令、结果如何、未覆盖项是什么

## 6. 严重级别定义

- `P0`：会导致错误结果被当成正确结果，或数据污染持续扩散
- `P1`：显著影响准确率/稳定性，但有兜底
- `P2`：可维护性、可观测性、可扩展性问题

## 7. Claude 执行约束

- 先写最小复现测试，再改实现。
- 每次只修一个主题，不做大杂烩提交。
- 不改变 Excel 输出契约，不改用户可见字段语义。
- 每次改动后至少运行：
  - `python tools/system_health_check.py --mode quick`
  - 受影响测试文件


- 任务3：门禁脚本 review 预检可配置化（避免地址写死）
