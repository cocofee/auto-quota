# 主材标准化与查价系统重构方案

日期：2026-04-16

## 1. 目标

本次不再继续对单个错误价格做零散修补，而是把“填主材名称”和“主材查价”重构成一套完整体系。

目标只有三个：

1. 先把待查主材整理成足够准确的标准化对象。
2. 再按统一规则查信息价和市场价，最终只选一条真实报价，不做平均值输出。
3. 让结果可追踪、可人工复核、可持续学习，而不是一次性碰运气。

## 2. 现状问题

当前系统已经有一层“填主材”逻辑，但输出仍然过于简单，主要只有：

- `suggested_name`
- `suggested_spec`

这会直接带来几个系统性问题：

1. 关键规格丢失。
示例：`De32*1.6MPa` 被缩成 `De32`，压力等级在后续查价中消失。

2. 分类边界太弱。
示例：`De63` 容易错匹配成 `De630`；`管`、`管件`、`阀门`、`设备`、`配件` 没有稳定分层。

3. 查价对象不完整。
仅用“名称 + 规格”查价，不足以区分材质、连接方式、系统属性、用途属性、对象类型。

4. 候选选择策略不稳定。
当前链路里仍存在“候选池打分后取代表价”的思路，容易把不该选的候选拿来当结果。

5. 结果可解释性不足。
前端能展示价格和链接，但不能清楚告诉用户“系统究竟按什么对象去查、为什么选中这一条”。

## 3. 用户确认的硬规则

以下规则视为本次重构的固定约束：

1. 有信息价先用信息价，没有再查市场价。
2. 广材网优先本省，没有再查全国，不依赖城市。
3. 规格必须强隔离。
   - `De63` 不能命中 `De630`
   - `DN100` 不能命中 `DN1000`
4. 不能输出平均价、中位价之类的统计价，最终必须落一条真实候选报价。
5. 允许“接近结果先填”，但不能出现明显荒谬价格。
6. `管` 和 `管件` 必须强分离，不能混选。
7. `管件` 在确实没有可靠价格时，可以回退为对应管材价格的 `70%`。
8. 不能把材料体系做窄，只支持 `管/管件/阀门` 不够，必须支持更广泛材料对象。
9. 填主材阶段必须尽量把关键信息抽出来，否则后面查价必然继续错。

## 4. 总体架构

重构后的主材系统分成六段：

1. 原始行解析
2. 主材标准化
3. 查价对象构建
4. 候选召回
5. 硬过滤与单条报价选择
6. 写回、导出、人工反馈

对应链路如下：

```text
Excel/任务原始主材行
  -> 清单上下文解析
  -> 主材标准化对象 normalized_material
  -> 查价查询对象 query_object
  -> 信息价候选召回
  -> 市场价候选召回
  -> 候选硬过滤
  -> 选择一条真实报价
  -> 写回价格/来源/链接/命中说明
  -> 人工纠正反馈
```

核心原则：

- 上游标准化优先于下游猜价格。
- 先过滤，再选择，不允许靠均值掩盖错匹配。
- 先判断“是不是同一种东西”，再判断“价格像不像”。

## 5. 主材标准化对象设计

现有 `suggested_name + suggested_spec` 不够，需要升级为结构化对象。

建议新增统一对象 `normalized_material`：

```json
{
  "raw_material_name": "",
  "raw_material_spec": "",
  "bill_name": "",
  "bill_desc": "",
  "main_name": "",
  "main_spec": "",
  "critical_spec": [],
  "material_tokens": [],
  "object_tokens": [],
  "shape_tokens": [],
  "connection_tokens": [],
  "system_tokens": [],
  "pressure_tokens": [],
  "unit": "",
  "family": "",
  "object_type": "",
  "is_installation_item": false,
  "is_accessory_like": false,
  "normalization_confidence": ""
}
```

字段说明：

- `main_name`：最终主查名称，例如 `滴水管PE`、`黄铜可调式减压阀`
- `main_spec`：主规格，例如 `De16`、`DN32`
- `critical_spec`：不能丢的关键限定，例如 `1.6MPa`、`PN16`、`S5`、壁厚、系列号
- `material_tokens`：材质类词，例如 `PE`、`HDPE`、`PPR`、`碳钢`、`黄铜`
- `object_tokens`：对象词，例如 `泄水阀`、`地漏`、`三通`、`传力接头`
- `shape_tokens`：形态词，例如 `90°弯头`、`异径三通`、`双法兰`
- `connection_tokens`：连接方式，例如 `热熔`、`法兰`、`螺纹`、`承插`
- `system_tokens`：用途或系统，例如 `给水`、`排水`、`灌溉`
- `family`：材料大类，不是窄分类，而是用于硬过滤的稳定边界
- `object_type`：对象层级，用于区分“管材本体 / 管件 / 阀门 / 接头 / 洁具 / 设备 / 配件 / 安装项”

## 6. 填主材算法重构

### 6.1 输入

输入不应只看主材列本身，而应联合以下信息：

- 主材原名称
- 主材原规格
- 清单名称
- 项目特征描述
- 单位
- 所在分部上下文

### 6.2 处理步骤

1. 规范化文本
   - 中英文符号统一
   - 规格分隔符统一
   - `De 63`、`DE63`、`de63` 归一
   - `1.6MPa`、`PN16` 等压力表达归一

2. 抽取键值对字段
   - `材质`
   - `规格`
   - `型号`
   - `压力等级`
   - `连接形式`
   - `名称`
   - `类型`
   - `介质`
   - `安装部位`

3. 切分主规格与关键规格
   - `De32*1.6MPa` -> `main_spec=De32`，`critical_spec=[1.6MPa]`
   - `De25 S5` -> `main_spec=De25`，`critical_spec=[S5]`
   - `DN700，P=1.0MPa` -> `main_spec=DN700`，`critical_spec=[1.0MPa]`

4. 提取对象类型
   - 先判断对象是什么，再拼名称
   - 例如 `泄水阀`、`过滤器`、`地漏`、`三通`、`传力接头`

5. 提取材质和系统词
   - 例如 `给水管HDPE` 应拆为 `system_tokens=[给水]`、`material_tokens=[HDPE]`

6. 生成主查名称
   - 对安装项名称，不直接拿安装项去查价
   - 例如 `绿地灌溉管线安装` 应细化成 `滴水管PE`

7. 生成主查规格
   - 规格严格保留主边界
   - 关键规格作为附加过滤条件保留，不允许丢

8. 给出标准化置信度
   - 高：名称、主规格、关键规格、对象类型都抽到
   - 中：名称、主规格抽到，关键规格缺失
   - 低：只能抽到部分字段，需要后续查价保守处理

### 6.3 输出要求

对前端不能只返回 `suggested_name` 和 `suggested_spec`，建议增加：

- `normalized_name`
- `normalized_spec`
- `normalized_query_text`
- `normalization_notes`
- `normalization_confidence`
- `critical_spec_text`
- `object_type`
- `family`

这样前端可以让用户一眼看到系统实际要查的对象是什么。

## 7. 查价查询对象设计

标准化对象不能直接原样拿去搜，需要再构建一层 `query_object`。

建议结构：

```json
{
  "query_name": "",
  "query_spec": "",
  "query_keywords": [],
  "must_match_tokens": [],
  "must_not_match_tokens": [],
  "source_order": ["info", "market"],
  "region_order": ["province", "national"],
  "family": "",
  "object_type": "",
  "unit": "",
  "fallback_strategy": ""
}
```

其中：

- `query_name`：主查名称
- `query_spec`：主规格 + 可拼接关键规格
- `query_keywords`：用于召回的不同搜索短语
- `must_match_tokens`：候选必须包含的词
- `must_not_match_tokens`：候选出现即判冲突的词

示例：

`滴水管PE + De16 + 1.6MPa + 热熔连接`

可构造成：

- `query_name = 滴水管PE`
- `query_spec = De16`
- `query_keywords = ["滴水管PE De16 1.6MPa", "滴水管PE De16", "PE滴水管 De16"]`
- `must_match_tokens = ["De16"]`
- `must_not_match_tokens = ["De160", "De16x16三通", "阀门"]`

## 8. 查价算法

### 8.1 查价顺序

固定顺序如下：

1. 本省信息价
2. 全国信息价
3. 本省市场价
4. 全国市场价
5. 特定回退策略

说明：

- 如果用户选了省份，只先查该省，不查城市。
- 广材网部分同样按“本省 -> 全国”。
- 信息价与市场价应拆开查询，不要混在一锅里做综合平均。

### 8.2 候选召回

候选召回允许多组查询词，但必须有层次：

1. 强查询
   - `主查名称 + 主规格 + 关键规格`
2. 标准查询
   - `主查名称 + 主规格`
3. 弱查询
   - `主查名称`
4. 同义扩展查询
   - `材质 + 对象词 + 主规格`

弱查询只能用于补召回，不能直接决定最终价格。

### 8.3 候选硬过滤

硬过滤必须先于价格比较执行。

#### 规则 A：规格边界强过滤

- `De63` 不匹配 `De630`
- `DN100` 不匹配 `DN1000`
- `Φ25` 不匹配 `Φ250`
- 主规格命中必须是完整边界，不允许子串误命中

#### 规则 B：对象类型强过滤

- 管材不能和管件混选
- 阀门不能和管件混选
- 设备不能和配件混选
- 洁具不能和阀门混选
- 安装项不能直接当材料报价

#### 规则 C：材质一致性过滤

- `HDPE` 不能随意命中 `UPVC`
- `黄铜` 不能命中 `铸铁`
- `碳钢` 不能命中 `不锈钢`

#### 规则 D：系统冲突过滤

- `给水` 与 `排水` 冲突时降级或剔除
- `灌溉` 与 `消防` 冲突时剔除

#### 规则 E：单位兼容过滤

- `m`、`个`、`套`、`kg`、`台` 需按兼容规则分组
- 单位明显不兼容时直接剔除

#### 规则 F：关键规格过滤

- 压力等级、系列号、壁厚等属于关键规格
- 候选不满足关键规格时不能算高置信候选

### 8.4 价格异常过滤

不使用全局平均价，也不使用全库历史均值去“修正”最终结果。

正确方式是：

1. 先得到“同一批候选”
2. 只在这一批候选内部判断价格带
3. 明显跳出该批候选价格带的候选剔除

例如：

- 同批候选大多在 `90 ~ 160`
- 某条候选是 `3489`
- 则 `3489` 应视为异常，不参与最终选择

但这个异常过滤只用于剔除荒谬候选，不能直接生成一个均值价格。

### 8.5 最终选价

最终结果必须是一条真实候选，不是统计结果。

选择顺序：

1. 是否同对象
2. 是否同主规格
3. 是否满足关键规格
4. 是否同材质
5. 是否同单位
6. 是否在合理价格带内
7. 是否为本省优先来源
8. 是否为信息价优先来源

选价输出必须携带：

- `selected_price`
- `selected_source`
- `selected_region`
- `selected_match_label`
- `selected_lookup_url`
- `selected_reason`

## 9. 回退策略

### 9.1 信息价无结果时

进入市场价查询，但保留“信息价未命中”的状态。

### 9.2 管件价格回退

仅在以下条件同时满足时启用：

1. 对象类型明确为管件
2. 没有找到可靠管件价格
3. 能找到同材质、同规格、同系统的对应管材价格

回退公式：

`pipe_fitting_price = pipe_price * 0.7`

并强制标记：

- `lookup_source = 回退估价`
- `selected_reason = 管件缺价，按对应管材70%回退`

### 9.3 允许近似结果先填

允许近似，不等于允许乱填。

近似结果至少要满足：

- 主规格严格一致
- 对象类型不冲突
- 材质不冲突
- 价格不异常

否则宁可留空，交给人工。

## 10. 前端与导出改造

前端需要从“只显示结果”改成“显示系统是怎么查到这个结果”。

建议表格新增或调整以下字段：

1. 标准化主材名称
2. 标准化规格
3. 关键规格
4. 对象类型
5. 匹配来源
6. 价格
7. 命中说明
8. 查价链接

链接列不要孤立放到最后，建议放在“价格说明/命中说明”后面，便于用户边看结果边点开核对。

导出 Excel 时建议同时写回：

- 价格
- 来源
- 链接
- 命中摘要

这样导出文件本身就具备复核线索。

## 11. 人工反馈闭环

后续算法必须可学习，因此要把人工纠正结构化沉淀。

建议新增反馈类型：

1. 主材标准化错误
2. 对象类型错误
3. 规格边界错误
4. 材质识别错误
5. 价格候选选错
6. 价格异常未拦截
7. 地区优先级错误

每次人工修正都应尽量记录：

- 原始主材
- 标准化对象
- 被选候选
- 正确候选
- 错误原因标签

## 13. 典型样例

### 样例 1

原始：

- 主材：`绿地灌溉管线安装`

- 连接形式：`热熔连接`

标准化后：

- `main_name = 滴水管PE`
- `main_spec = De16`
- `critical_spec = [1.6MPa]`
- `connection_tokens = [热熔]`
- `object_type = 管材`

### 样例 2

原始：

- 主材：`给水管HDPE`

过滤要求：

- 绝对不能命中 `De630`

### 样例 3

原始：

- 主材：`泄水阀`
- 规格：`De63`

候选：

- 候选 A：`128.82`

- B 应被视为异常候选
- 最终落 A，而不是做平均

### 样例 4

标准化后：

- `object_type = 管件/接头`

## 14. 实施原则

1. 先重构标准化，再重构查价，不反过来。
2. 先补结构化测试，再改主逻辑。
3. 不为某个材料写单点特判，尽量抽象成通用规则。
4. 不再用平均值掩盖错误选价。
5. 对无法证明正确的结果，宁可降级为低置信度或留空。

## 15. 本次结论

后续真正要做的不是“修一个 63/630 的 bug”，而是把整条链路改成：

`原始主材 -> 结构化标准化对象 -> 分层查价 -> 硬过滤 -> 选一条真实报价 -> 带解释写回`

只有这样，主材名称、规格、地区、来源、价格选择才能同时稳定下来。
