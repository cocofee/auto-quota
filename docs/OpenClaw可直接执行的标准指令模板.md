# OpenClaw 可直接执行的标准指令模板

## 1. 用法

这份文档不是给你看的理论说明，而是给你直接复制发给 OpenClaw 的。

你以后不要自己临时组织话术，直接从这里复制一段发给它。

---

## 2. 首次接管项目时的总指令

适合场景：

- 第一次把项目交给 OpenClaw
- 让它进入全权负责模式
- 让它知道权限边界

直接发下面这段：

```text
按以下文档执行：
1. docs/OpenClaw全权负责模式实施方案.md
2. docs/Jarvis快速提效实施方案.md

当前项目根目录：
C:\Users\Administrator\Documents\trae_projects\auto-quota

你的权限范围：
- 可读整个项目代码
- 可修改 src、web、tests、tools、docs
- 可运行本地测试、构建、打包、预览验证
- 可生成 commit
- 不可直接修改正式生产数据库
- 不可直接跳过验证发布正式环境

你的工作方式：
- 直接分析问题
- 直接修改代码
- 直接补测试
- 每做一步先验证，再继续下一步
- 输出验收方法
- 如果验证不通过，不要继续下一步

当前目标：
进入 OpenClaw 全权负责模式，后续由你负责 Jarvis 的持续优化，我只负责确认结果。
```

## 4. 让它修“错大类”问题的指令

适合场景：

- Jarvis 方向对，但经常错大类
- 例如法兰阀门跑成塑料阀门

直接发下面这段：

```text
分析最近人工审核中“错大类”的高频样本，重点处理 Jarvis 把对象分错大类的问题。

要求：
1. 找出最高频的 3 类错大类问题
2. 判断问题来自 query、候选召回、重排还是 review 规则
3. 直接修改相关代码
4. 补充对应测试
5. 跑验证并告诉我修复前后差异

优先关注：
- 阀门类
- 法兰类
- 塑料管类
- 安装方式混淆类

不要只给建议，直接执行。
```

```text
排查并修复 OpenClaw 审核流程不生效的问题。

目标：
1. 找出当前审核建议被拦截的真实原因
2. 判断是红灯限制、权限判断、状态流转还是保存逻辑问题
3. 直接修代码
4. 补测试
5. 给出验收步骤

要求：
- 不要只停留在接口层
- 要覆盖前端显示、后端规则、状态保存、再次进入任务后的结果恢复
- 修完后告诉我前端在哪里能看到效果
```

## 6. 让它修人工审核页面的指令

适合场景：

- 页面太复杂
- 不能一行内处理完
- 候选选择不顺手

直接发下面这段：

```text
优化人工审核页面，目标是一行内完成审核决策，不要让我上下翻动或点很多层。

必须完成：
1. 一行内同时看到 Jarvis 定额、OpenClaw 审核定额、人工审核定额
2. 人工审核列可以直接从候选里点选
3. 也可以直接搜索正确定额
4. 标题行固定，不随滚动移动
5. 列标题和内容必须严格对齐
6. 审核结果保存后，再次进入任务不能丢失

要求：
- 直接改前端和后端
- 改完后告诉我如何验收
- 不要做成卡片堆叠式页面
- 以表格一行处理完为核心目标
```

## 7. 让它自动分析失败样本并修复的指令

适合场景：

- 你不想自己盯错单
- 希望它自己找问题自己修

直接发下面这段：

```text
进入自动优化模式。

任务：
1. 收集最近一批人工改过的失败样本
2. 按错因聚类
3. 找出最值得先修的 1 到 2 类问题
4. 直接改代码
5. 补测试
6. 跑验证
7. 输出修复结果和下一步建议

要求：
- 一次只修一个问题域
- 每次改动都必须可验证
- 不要大范围同时重构多个模块
- 如果收益不明显，停止扩散修改
```

---

## 8. 让它读代码并自己改的指令

适合场景：

- 你已经明确允许它直接改仓库

直接发下面这段：

```text
你现在有权限直接读取并修改本项目代码。

工作要求：
1. 先定位问题所在模块
2. 只修改必要文件
3. 修改后补测试
4. 跑本地验证
5. 输出修改结果、测试结果、验收方法

边界要求：
- 不要直接改正式生产数据
- 不要跳过验证
- 不要无理由大范围重构
- 如果发现多个问题，按优先级一个一个处理
```
