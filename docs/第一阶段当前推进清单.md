## 进行中

### 0. 正式确认主链已查清性能瓶颈，需继续产品化收口

今天联调新增确认：`review-confirm` 过去看起来卡顿，并非请求体或鉴权主因，而是正式纠正后同步等待经验库/向量写回，导致接口返回体验不稳定。日志已看到：

- `向量模型加载完全失败`
- `向量索引添加失败: 'NoneType' object has no attribute 'encode'`

已在 `web/backend/app/api/shared.py` 做最小解耦：`store_experience()` / `store_experience_batch()` 改为后台 best-effort 执行，失败仅记日志，不阻塞主接口。当前结果：解耦后 `review-confirm` 已连续多条 `200 OK` 成功。

下一步要收口的，不再是“能不能 confirm”，而是：

- 前端确认后状态刷新是否及时
- 批量/连续确认体验是否稳定
- 人工确认后的回流学习是否需要进一步异步化、监控化

### 1. 绿黄红分流还没完全收口到“主流程口径”

当前已确认前后端阈值一致：

- 后端：`GREEN_THRESHOLD = 90`、`YELLOW_THRESHOLD = 75`
- 前端：`GREEN_THRESHOLD = 90`、`YELLOW_THRESHOLD = 75`
- 前后端都优先读取已有 `light_status`，缺失时再按分数回退计算

已确认结论：当前不存在明显的前后端灯色阈值分叉。

但还需要继续确认：

- 主链每个入口是否统一写入 `light_status`
- 报表 / 审核页 / 自动审核任务是否全部优先使用该字段
- 自动审核是否完全避免碰 `green`

### 2. 人工错因回流虽然能写，但还没形成统一运营口径

当前 `review-confirm` 已支持：

- `human_feedback_payload`

但还缺：

- 推荐的固定错因字段模板
- 页面/批量工具的统一写法
- 后续统计和知识分层的默认消费规则
