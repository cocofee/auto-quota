# OpenClaw 全权负责模式实施方案

## 1. 目标

这份方案的目标很简单：

你不会编程，也不需要学会改代码。

以后由 OpenClaw 负责分析问题、修改代码、跑验证、准备发布，你只负责：

- 下业务指令
- 看结果
- 决定是否确认上线

也就是把 OpenClaw 变成你的自动工程师，而不是单纯的审核助手。

## 2. 什么叫全权负责

这里的全权负责不是无限制权限。

合理的全权负责，应该是：

- 它全权负责开发和优化
- 它全权负责本地验证和测试
- 它可以全权负责生成部署包
- 它可以全权负责提交代码
- 它不直接全权负责正式生产发布

原因很现实：

如果它直接无条件改生产结果、改线上数据库、直接覆盖正式版本，一旦改错，你没有能力兜底。

所以最适合你的模式是：

开发全权交给 OpenClaw  
上线确认保留给你

## 3. 最适合你的权限分层

建议把权限分成 3 层。

### 第一层：本地开发权限

这一层可以完全放给 OpenClaw。

包含：

- 读取整个仓库代码
- 修改白名单目录
- 新增测试
- 跑本地测试
- 跑 benchmark
- 分析失败样本
- 自动生成修复 commit
- 自动生成说明文档

### 第二层：测试环境权限

这一层也可以基本放给 OpenClaw。

包含：

- 启动本地服务
- 生成预览版本
- 部署到测试环境
- 自动跑一轮冒烟验证
- 导出结果给你验收

### 第三层：正式环境权限

这一层不要完全放开。

正式环境建议保留：

- 由 OpenClaw 准备发布内容
- 由你确认是否发布

也就是：

OpenClaw 可以做到随时可发  
但最后点发布的动作由你确认

## 4. 建议给 OpenClaw 的实际权限

### 4.1 代码权限

建议先开放这些目录：

- `src/`
- `tests/`
- `tools/`
- `web/backend/app/`
- `web/frontend/src/`
- `docs/`

其中最关键的是：

- `src/query_builder.py`
- `src/query_router.py`
- `src/match_pipeline.py`
- `src/review_checkers.py`
- `src/review_correctors.py`
- `web/backend/app/api/openclaw.py`
- `web/backend/app/services/openclaw_review_service.py`
- `web/frontend/src/pages/Results/`

### 4.2 数据权限

建议分开给。

可以给：

- 读取任务结果
- 读取候选定额
- 读取人工审核记录
- 读取失败样本包
- 写入建议草稿
- 写入测试数据

不要直接给：

- 直接修改正式审核结果
- 直接覆盖正式历史数据
- 直接删除结果数据

### 4.3 命令权限

可以给它执行：

- 本地测试
- benchmark
- 构建前端
- 构建后端
- 生成部署包
- 本地部署

不要默认给它执行：

- 直接删除数据库
- 强制回滚生产数据
- 大范围清空目录

## 5. 最适合你的工作方式

你以后不要再对它说“帮我看看代码”这种太泛的话。

你应该直接下业务命令。

## 6. 你以后可以直接这样对 OpenClaw 说

### 6.1 修正错误类

- “分析最近 100 条人工改过的错单，找出最高频 3 类错因，直接修代码并补测试。”
- “把阀门、法兰、塑料管这三类误判率降下来，改完以后跑验证给我结果。”
- “把红灯项为什么一直不能提交审核建议查出来，修到能走通，再给我验收步骤。”

### 6.2 优化系统类

- “根据最近一周人工审核结果，优化 Jarvis 的检索和重排，优先减少错大类。”
- “把 OpenClaw 二审建议的采纳率提高，重点优化黄灯项。”
- “把候选池统一，让 Jarvis、OpenClaw、人工审核说的是同一批候选。”

### 6.3 页面和流程类

- “把人工审核页面改成一行内处理完，不要上下翻。”
- “把人工审核支持从候选直接选定额，也支持搜索正确定额。”
- “把审核结果保存好，再次进入任务时不要丢状态。”

### 6.4 自动优化类

- “每天把失败样本自动汇总给 OpenClaw，分析错因并生成第二天的修复建议。”
- “跑完 Jarvis 后自动触发 OpenClaw 做二审，只处理黄灯和红灯项。”
- “让 OpenClaw 对最近失败样本自动出 patch，但必须附带测试。”

## 7. OpenClaw 应该自动执行的标准流程

你给出命令后，它应该按这个顺序自己执行：

1. 读取相关任务、结果、失败样本
2. 归类问题和错因
3. 判断是检索问题、排序问题、规则问题还是页面问题
4. 只修改相关代码
5. 自动补测试
6. 跑本地验证
7. 输出修复结果
8. 生成 commit 或部署包
9. 等你确认

也就是说，你不需要参与中间的技术过程。

## 8. 最适合的自动化边界

建议把 OpenClaw 的自动化做成两种模式。

### 模式 A：自动修复模式

适合：

- 小范围规则修复
- 页面交互优化
- 查询词修补
- 候选排序优化
- 测试补充

这一类可以让它自动改、自动测、自动提交。

### 模式 B：自动建议模式

适合：

- 大范围重构
- 排序主链路替换
- 数据结构改造
- 正式发布前变更

这一类让它只出方案、出 patch、出测试、出验证结果，最后由你点确认。

## 9. 你最该保留的唯一人工动作

你不会编程，所以你不应该去审核代码。

你真正该保留的人工动作只有一个：

看业务结果是否对

也就是：

- 页面是不是更顺手了
- 审核是不是更快了
- 错单是不是变少了
- OpenClaw 的建议是不是更可信了

你不需要判断代码写得漂不漂亮，只判断业务结果。

## 10. 推荐的实际落地方式

最推荐你按下面方式落地。

### 第一步

先给 OpenClaw 本地开发全权：

- 允许读代码
- 允许改代码
- 允许跑测试
- 允许出 commit

### 第二步

再给 OpenClaw 测试环境全权：

- 允许启动服务
- 允许打包
- 允许部署测试版
- 允许自动验证主要流程

### 第三步

最后保留正式发布确认：

- OpenClaw 准备好发布内容
- 你只看结果并确认

## 11. 最适合你的最终形态

最终建议的工作分工如下：

- Jarvis：负责初审和候选生成
- OpenClaw：负责二审、修系统、提 patch、跑验证
- 你：负责看业务结果并确认发布

这才是真正适合不会编程的管理方式。

## 12. 一句话结论

可以让 OpenClaw 基本全权负责开发和优化。

但不要让它无条件全权控制生产发布。

最好的模式是：

OpenClaw 负责做  
你负责确认

这样效率最高，也最稳。
