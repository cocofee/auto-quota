## 1. 目标

把 Jarvis 从"单机可跑"升级为"可量化、可回归、可多用户部署"的工程系统。

四个核心目标：
1. 准确率稳定提升（不是偶然提升，有benchmark回归保护）
2. 运行稳定性提升（网络差/模型缺失也能可控降级）
3. 多用户隔离安全（无串数据、无互相污染）
4. 发布可控（每次升级都有验收和回滚）

---

## 2. 执行总规则（Claude必须遵守）

1. 严格按 L1 → L2 → L3 → L4 → L5 → L6 顺序执行。
2. 每次只改一个阶段，不跨阶段混改。
3. 每阶段先补最小复现测试，再修实现。
4. 每阶段结束交 3 个文件：findings / patch_summary / validation。
5. 每阶段必须给"回滚点"与"残余风险"。
6. 遵守项目 CLAUDE.md 中的执行协议（质量门禁、放行条件、问题等级）。

### L2 正确性层（匹配质量）

目标：绿率从84%提升到88%+，红率从14%降到8%以下。

#### L2-a：建立固定benchmark + 采集基线

必须先有量化基线，否则后续改进无法衡量。

任务：
1. 选定4组固定benchmark数据集（参见第4节基线指标）
2. 对每组跑一次Agent模式，记录Top1命中率、绿率、红率、fallback占比
3. 将结果固化为 `tests/benchmark_baseline.json`，后续每次改动都回归

关键文件：
- `tools/eval_vs_human.py`：人工对比评测工具
- `src/accuracy_tracker.py`：准确率追踪（run_history + review_history表）
- `tests/eval_golden.py`：黄金集评测

#### L2-b：参数提取率提升（58% → 75%+）

当前 `bill_cleaner.py` 只能从58%的清单项提取结构化参数（DN/截面/材质等），未提取参数的清单项参数验证无效，直接退化为纯搜索排序。

任务：
1. 统计哪些类型清单项参数提取失败（按专业分类）
2. 补充 `text_parser.py` 中缺失的参数模式
3. 改进 `bill_cleaner.py` 中特征描述的解析逻辑

关键文件：
- `src/text_parser.py`：参数提取（DN、截面、材质、连接方式等）
- `src/bill_cleaner.py`：清单清洗（名称修正+专业分类+参数提取）
- `data/dict/engineering_dict.txt`：专业词典

#### L2-c：低准确率清单类型专项优化

根据 `output/full_comparison.txt` 的分类统计，以下类型准确率为0%，需专项优化：

| 类型 | 数量 | 问题 |
|------|------|------|
| 配电箱 | 47条 | query构建未区分箱体安装vs箱内配线 |
| 配管 | 47条 | 材质代号(SC/PC/CT)→中文名称映射不全 |
| 配线 | 27条 | 导线型号→截面+穿线方式提取不准 |
| 电力电缆 | 15条 | 敷设方式识别不够 |
| 装饰灯 | 14条 | 灯具类型映射不覆盖 |

关键文件：
- `src/query_builder.py`：搜索查询构建（build_quota_query，第285-531行）
- `src/specialty_classifier.py`：专业分类
- `data/engineering_synonyms.json`：同义词替换

验收标准：
1. 固定benchmark的绿率 ≥ 88%
2. 红率 ≤ 8%
3. 已有105个测试全通过（不退化）
4. 低准确率类型至少提升到50%+

### L3 检索与重排层（效率+召回）

目标：召回率不低于基线，搜索模式准确率提升。

#### L3-a：Query重写策略优化

当前 `query_builder.py` 的规范化有时过度替换（如灯具类的复杂映射），导致搜索方向偏移。

任务：
1. 审计当前query重写规则，标记过度替换的case
2. 对"误归类"的case补充回归测试
3. 优化同义词替换策略（`engineering_synonyms.json`），避免过拟合

#### L3-b：无模型降级策略加固

向量模型或重排模型不可用时，系统应该有明确的降级策略，而不是静默返回差结果。

任务：
1. 验证无向量模型时BM25单路搜索的质量（benchmark对比）
2. 验证无重排模型时的候选排序质量
3. 确保降级路径有明确日志和告警

关键文件：
- `src/hybrid_searcher.py`：混合搜索（BM25+向量，RRF融合）
- `src/reranker.py`：重排模型
- `src/model_cache.py`：模型缓存与降级统计

验收标准：
1. 召回率不低于L2基线
2. 无模型场景可稳定跑完，有降级日志
3. `tools/system_health_check.py --mode full` 通过

### L4 审核与纠错层（安全落盘）

目标：自动纠错误改率降到2%以下，多Sheet回写无错位。

#### L4-a：自动纠错二次验真

当前 `jarvis_auto_review.py` 对候选直接替换，无二次校验。

任务：
1. 纠错前对替换定额做参数一致性检查
2. 高风险纠错（置信度变化大、跨专业替换）转人工标记
3. 纠错结果标记来源追踪（auto_review vs manual_correction）

#### L4-b：回写定位策略加固

当前 `output_writer.py` 已有三级回退（seq→bill_code→顺序），但需要更多边界保护。

任务：
1. 验证多Sheet文件的回写准确性
2. 验证合并单元格场景的回写
3. 补充回写定位失败时的告警日志

关键文件：
- `tools/jarvis_auto_review.py`：自动审核
- `tools/jarvis_correct.py`：纠正Excel
- `src/output_writer.py`：结果Excel输出

验收标准：
1. 纠错链路端到端测试通过
2. 含多Sheet的真实样本回归通过
3. 落库数据可追踪来源与版本

### L5 多用户与部署层（服务化）

目标：5路并发稳定运行，成功率≥99%。

L1已完成请求级隔离基础，L5在此基础上做压测和服务化。

#### L5-a：并发压测

任务：
1. 编写并发压测脚本（5路×相同输入 + 5路×不同输入）

3. 输出压测报告（成功率/耗时/P95/文件数量校验）

#### L5-b：服务接口化（可选，看需求）

任务：
1. 将 `pipeline()` 封装为API接口（FastAPI或Flask）
2. 添加任务队列（避免CPU/GPU竞争）
3. 用户与任务隔离（run_id绑定用户ID）

验收标准：
1. 并发5路稳定运行，成功率≥99%
2. 无输出覆盖、无日志串线
3. 锁冲突可自动重试，不丢任务
