# 个人开发版详细实施方案

## 1. 前提和目标

这份方案按下面几个现实前提来设计：

1. 你是个人开发，不是团队协作
2. 当前系统已经有一套可用的 `套定额` 主干
3. 不做图纸识别
4. 真实业务入口不是固定格式，可能是 `Excel / Word / PDF / 图片 / 半结构化文本`
5. 目标不是做一个“大而全平台”，而是做一个能持续闭环、能每天进步的个人系统

所以整体策略必须是：

`在现有套定额主干上，补一个轻量但稳定的资料归一化 + 编清单中台，再补主材和综合单价检查，不搞过早复杂化`

---

## 2. 个人开发的总原则

### 2.1 架构原则

对你现在最合适的架构不是微服务，而是：

`模块化单体`

原因很简单：

1. 你一个人维护，模块边界重要，但服务拆分成本太高
2. 现在真正稳定的核心还是本地 Python 匹配链路
3. 一旦拆太散，调试、部署、日志、状态同步会比算法本身更耗时间

建议保留现在这个形态：

1. 一个主代码仓
2. 一套 Python 核心引擎
3. Web 后端只做任务编排、状态存取、文件上传和结果展示
4. 真正的业务决策都沉到 `src/`

也就是说：

`web/backend` 是壳

`src/` 才是业务内核

### 2.2 产品原则

先做“可审核、可纠错、可回写”，再做“全自动”。

对造价场景来说，最值钱的不是“100% 自动”，而是：

1. 自动做 70%
2. 把剩下 30% 变成可快速审核
3. 审完之后系统能真正学进去

### 2.3 开发原则

每做一层，都必须满足三件事：

1. 能单独评测
2. 能单独回放
3. 能单独人工修正

如果某一层只能“跑过去看结果”，不能独立评测和回放，那层后面一定会失控。

## 4. 最终目标架构

对个人开发，建议只保留 6 个核心模块，不要更多。

### 4.1 六个核心模块

1. 输入适配器
2. 编清单中台
3. 套定额引擎
4. 主材推荐器
5. 综合单价检查器
6. 学习与评测中心

整体流程：

`原始文件 -> 输入适配 -> 编清单标准化 -> 套定额 -> 主材推荐 -> 综合单价检查 -> 人工审核 -> 回写学习`

### 4.2 服务形态

建议保持三层，不再增加服务：

1. `src/`
   核心业务层
2. `web/backend`
   API、任务状态、文件管理
3. `web/frontend`
   上传、预览、审核、下载

不要为了“看起来高级”拆成更多进程或更多服务。

## 6. 编清单中台怎么做

这是第一优先级。

### 6.1 为什么它最重要

因为你现在的入口并不稳定，而且不只是 Excel。

如果入口文本脏、结构乱、单位乱、上下文丢失，后面的：

1. 搜索 query 会偏
2. 专业判断会偏
3. 定额匹配会偏
4. 主材也会偏
5. 价格检查更会偏

所以最先补的是：

`把各种乱资料变成稳定的标准 bill_item`

### 6.2 建议拆成 4 个文件

建议新增：

1. `src/input_adapters/excel_adapter.py`
2. `src/input_adapters/word_adapter.py`
3. `src/input_adapters/pdf_adapter.py`
4. `src/input_adapters/image_adapter.py`
5. `src/bill_structure_resolver.py`
6. `src/bill_compiler.py`

### 6.3 每个文件负责什么

#### excel_adapter.py

负责把 Excel 类资料转成统一的 `raw_item`。

重点做：

1. 表头识别
2. 列映射
3. 分部层级继承
4. 属性行归并
5. 原始字段保留

#### word_adapter.py / pdf_adapter.py / image_adapter.py

负责处理 Word、PDF、图片等非标准资料。

重点做：

1. 文字提取
2. 表格区域识别
3. 段落和表格混排处理
4. 把结果统一成中间 `raw_item`

#### bill_structure_resolver.py

负责判断一行到底是什么。

至少区分：

1. 清单主项
2. 描述属性行
3. 分部标题
4. 合计行
5. 措施项
6. 噪声行

#### bill_compiler.py

负责把 `raw_item` 编译成 `compiled_bill_item`。

做三件事：

1. 合并上下文
2. 标准化名称和描述
3. 生成后续可用于套定额的结构化结果

### 6.4 编清单不要追求什么

第一版不要追求：

1. 编清单和人工一字不差
2. 名称特别优美
3. 所有细节都补全

第一版只追求：

1. 专业不偏
2. 主项不丢
3. 单位不乱
4. 参数可提取
5. 能稳定喂给套定额

### 6.5 编清单验收标准

第一阶段验收只看 5 个指标：

1. `usable_rate >= 85%`
2. `specialty_correct_rate >= 95%`
3. `unit_normalization_rate >= 95%`
4. `main_item_capture_rate >= 95%`
5. `人工整条重编率 <= 15%`

这里最重要的是 `usable_rate`。

## 7. 套定额层怎么接

这一层你不要大改。

### 7.1 原则

现有套定额链路已经是你的核心资产，处理方式应该是：

`改输入，不改骨架`

### 7.2 接入方式

`bill_compiler -> bill_cleaner -> text_parser -> query_builder -> match_pipeline -> match_engine`

### 7.3 需要补的东西

这一层只补三件事：

1. 把 `compiled_bill_item.normalized` 映射成当前 `bill_cleaner` 可吃的格式
2. 在匹配结果里把 `compile_confidence` 一并保留
3. 每条结果都带完整 trace

建议让匹配结果至少多出这些字段：

```python
match_result = {
    "bill_id": "...",
    "compile_confidence": 0.92,
    "search_query": "...",
    "match_source": "experience | rule | search | agent",
    "quotas": [...],
    "confidence": 87.0,
    "trace": [...],
}
```

### 7.4 为什么要把 compile_confidence 带下去

因为以后你会遇到很多看起来像“套错定额”的问题，实际是“编清单先编偏了”。

如果没有这条链路，你根本分不清责任在哪一层。

主材层第一阶段不是“自动组主材”，而是：

1. 能挂上主材
2. 能发现明显错挂
3. 能积累主材事实

### 8.2 建议新增文件

1. `src/material_normalizer.py`
2. `src/material_recommender.py`
3. `src/material_checker.py`

### 8.3 推荐逻辑顺序

只做四级优先级，够了。

1. 输入文件自带主材

### 8.5 主材验收标准

1. `material_attach_rate >= 80%`

## 9. 综合单价检查器怎么做

这是第三优先级。

### 9.1 为什么先做检查器，不做自动组价

因为你是个人开发。

自动组价牵涉：

1. 取费口径
2. 人工材料机械拆分
3. 地区规则
4. 税费
5. 信息价与市场价混用

这个坑太大，不适合作为当前主战场。

但检查器很适合，因为：

1. 有历史项目
2. 有广联达导出结果
3. 有主材价格库
4. 业务价值直接

### 9.2 第一版只做 4 类风险

1. 无价
2. 离群价
3. 单位换算异常
4. 主材价格与综合单价明显不匹配

### 9.3 建议新增文件

1. `src/price_feature_builder.py`
2. `src/unit_price_checker.py`
3. `src/price_fact_store.py`

### 9.4 price_feature_builder 做什么

把每一条清单展开成价格特征。

例如：

price_features = {

## 11. 前端怎么配合

对个人开发，前端不要做复杂工作台。

只做一个够用的审核页。

### 11.1 审核页必须同时看到四块信息

1. 编清单结果
2. 套定额结果
3. 主材结果
4. 综合单价风险提示

### 11.2 审核动作必须能明确区分

人工修正时，必须知道自己改的是哪一层：

1. 改编清单
2. 改定额
3. 改主材
4. 标记价格误报

否则所有反馈最后都会变成一团，系统没法学习。

### 11.3 前端第一版功能

第一版就做这些：

1. 上传文件
2. 预览编清单结果
3. 查看每条定额匹配
4. 展示主材和价格风险
5. 人工修改并保存
6. 导出结果

1. `bill_compiler.py`
2. `project_pipeline.py`
3. `run_compile_benchmark.py`

因为这三个一出来，整个项目的“入口标准化”就开始成立了。

## 14. 你最该避免的坑

### 14.1 过早拆服务

你一个人开发，最怕不是代码不优雅，而是系统难以调试。

### 14.2 过早做全自动组价

这个战线太长，会把你拖进无穷口径细节。

### 14.3 没有统一中间对象

这会让你后面每个功能都要重新解释“清单项是什么”。

### 14.4 审核反馈没有分层

如果人工只改最终结果，不记录是编清单错还是定额错，系统永远学不快。

### 14.5 只看最终命中率

最终命中率高，不代表入口和主材层是健康的。

你一定要能单独看到：

1. 编清单错多少
2. 套定额错多少
3. 主材错多少
4. 价格检查误报多少

## 15. 个人开发版的建议节奏

如果按个人节奏，我建议按下面方式推进。

### 第 1 周

只做：

1. 定义统一对象
2. 做编清单 MVP
3. 跑 20~50 个真实样本

### 第 2 周

只做：

1. 接通现有套定额链
2. 做项目快照
3. 做人工审核保存

### 第 3 周

只做：

1. 做主材推荐 MVP
2. 主材写回与展示
3. 建立主材修正反馈

### 第 4 周

只做：

1. 做综合单价检查 MVP
2. 标记异常项
3. 开始积累 `unit_price_fact`

### 第 2 个月开始

再做这些长期项：

1. 编清单 benchmark 扩样
2. 主材模板学习
3. 价格异常阈值优化
4. 审核页体验优化

## 16. 最后的明确结论

对你这种“个人开发、已经有套定额基础、真实入口是各种乱资料”的情况，最正确的路线不是横向扩张，而是纵向打通。

具体就是：

1. 先做 `编清单中台`
2. 再做 `主材正式化`
3. 最后做 `综合单价检查器`

而且整个实现方式必须是：

`模块化单体 + 统一中间对象 + 分层评测 + 分层回写`

这条路线的好处是：

1. 适合一个人长期维护
2. 不会把系统复杂度推爆
3. 每一阶段都能马上产生业务价值
4. 每一阶段都能沉淀成下一阶段的数据基础

如果后面只允许你优先做一个模块，那就只做：

`bill_compiler`

因为它是你这整套闭环真正缺的那个中轴。
