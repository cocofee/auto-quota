# 编清单 -> 套定额 -> 主材 -> 综合单价检查闭环实施方案

## 1. 先把边界说死

这套方案不包含图纸识别，也不以图纸识别为前提。

当前真实入口只有两类：

1. `GQI` 导出的工程量报表 / 工程量清单 / 过程表
2. 不太规范的清单 Excel，需要先做清单标准化，再进入套定额

所以系统要补的不是“识图层”，而是一个放在现有匹配引擎前面的：

`编清单 / 清单标准化中台`

---

## 2. 结合现有代码，当前架构的真实状态

### 2.1 已经比较成熟的部分

1. `套定额主引擎`
   现有主干已经完整，包括 `src/bill_reader.py`、`src/bill_cleaner.py`、`src/text_parser.py`、`src/query_builder.py`、`src/match_pipeline.py`、`src/match_engine.py`。
2. `经验学习闭环`
   现有 `src/experience_db.py`、`src/universal_kb.py`、`tools/import_reference.py`、`tools/jarvis_store.py` 已经能把项目结果和人工修正沉淀回库。
3. `主材数据底座`
   `src/material_db.py` 已经不是临时脚本，而是清晰的三层结构：
   `material_master / price_fact / quota_material_observation`。
4. `结果输出`
   `src/output_writer.py` 已经能把定额行、主材行、附加信息写回 Excel，适合继续扩展。

### 2.2 还没有成为“主流水线”的部分

1. `编清单`
   目前更像“读清单 + 清洗 + 搜索文本构造”，不是一个独立的“编清单引擎”。
2. `自动上主材`
   当前更多是“从输入文件提取主材”和“从经验里回显主材”，还不是一个稳定的主材推荐决策层。
3. `综合单价检查`
   当前明确可见的是 `tools/price_backfill.py` 这类“把广联达已有价格回填到甲方原表”的工具，不是原生的综合单价检查引擎。

### 2.3 结论

现在最强的是：

`标准清单 -> 套定额 -> 审核纠偏 -> 学习`

现在最弱的是：

`GQI/非标输入 -> 编清单/标准化 -> 主材推荐 -> 综合单价检查`

所以后续建设不应该继续把所有资源压在“套定额更聪明”上，而应该先补齐前后两段。

---

## 3. 目标架构：五层闭环，不再是一条脚本链

建议把系统重构为五层：

1. 输入适配层
2. 编清单 / 清单标准化层
3. 套定额层
4. 主材层
5. 综合单价检查层

整体数据流：

`原始文件 -> 原始行解析 -> 编清单标准化 -> 标准 bill_item -> 套定额 -> 主材推荐/校验 -> 综合单价检查 -> 人工确认 -> 经验回写`

---

## 4. 建议中的目标模块划分

### 4.1 输入适配层

目标：先把不同来源统一成同一份中间结构，而不是让后面模块直接吃 Excel 行。

建议新增：

- `src/input_adapters/gqi_adapter.py`
- `src/input_adapters/nonstandard_bill_adapter.py`
- `src/input_adapters/compiled_bill_adapter.py`

统一输出结构：

```python
raw_item = {
    "source_type": "gqi_report | nonstandard_bill | standard_bill",
    "project_id": "...",
    "sheet_name": "...",
    "row_no": 123,
    "raw_code": "...",
    "raw_name": "...",
    "raw_desc": "...",
    "raw_unit": "...",
    "raw_qty": 12.3,
    "raw_price": None,
    "raw_total": None,
    "context_path": ["分部", "分项", "清单项"],
    "extra_fields": {...},
}
```

现有可复用能力：

- `src/bill_reader.py` 负责 Excel 行识别
- `src/output_writer.py` 反向写出时已经具备行级结构理解

这里不要再把 GQI、甲方清单、手工清单分别走三套逻辑。统一进入 `raw_item` 才能做后续评测和学习。

### 4.2 编清单 / 清单标准化中台

这是接下来最值得投入的核心。

建议新增：

- `src/bill_compiler.py`
- `src/bill_normalizer.py`
- `src/bill_structure_resolver.py`
- `src/bill_compile_evaluator.py`

职责拆开：

1. `bill_normalizer`
   负责名称修正、单位标准化、描述切分、空值补全、分部上下文继承。
2. `bill_structure_resolver`
   负责识别“这一行是独立清单项、属性行、说明行、附属行还是噪声行”。
3. `bill_compiler`
   负责把 GQI 行或非标行编译成标准 `bill_item`。
4. `bill_compile_evaluator`
   负责评测“编出来的清单项是否稳定、是否可用于套定额”。

标准输出建议定义为：

```python
compiled_bill_item = {
    "bill_id": "...",
    "source_type": "...",
    "origin_trace": {
        "sheet_name": "...",
        "row_span": [121, 122, 123],
        "raw_keys": ["项目名称", "特征描述", "单位"],
    },
    "name": "焊接钢管安装",
    "description": "沟槽连接 DN150 室内消火栓给水管",
    "unit": "m",
    "quantity": 86.4,
    "specialty": "安装",
    "book_hint": "C",
    "category_hint": "消防",
    "is_measure_item": False,
    "compile_confidence": 0.91,
    "compile_flags": ["unit_normalized", "context_inherited"],
    "source_materials": [],
    "source_prices": {
        "unit_price": None,
        "total_price": None,
    },
}
```

这里最关键的是：以后系统评测、经验学习、错因分析，都围绕 `compiled_bill_item` 做，而不是围绕原始 Excel 文本做。

### 4.3 套定额层

这一层以现有能力为主，不重写，只改接入口。

做法：

1. `bill_compiler` 的输出直接喂给现有 `bill_cleaner -> query_builder -> match_pipeline -> match_engine`
2. 套定额结果必须带回完整 trace，包括：
   - 原始行来源
   - 编清单置信度
   - 搜索 query
   - 经验命中 / 规则命中 / 检索命中来源
   - 人工修正后的最终版本

建议新增轻量编排器：

- `src/project_pipeline.py`

职责是把“编清单结果”和“套定额结果”串起来，形成统一项目状态。

### 4.4 主材层

主材要从“展示信息”升级为“决策结果”。

建议新增：

- `src/material_recommender.py`
- `src/material_normalizer.py`
- `src/material_checker.py`

三件事分开做：

1. `material_normalizer`
   把名称、规格、单位统一成主材标准词。
2. `material_recommender`
   根据 `quota + bill + province + 历史项目` 推荐主材。
3. `material_checker`
   做校验，不允许明显错挂。

主材推荐来源按优先级：

1. 输入文件原始主材
2. 历史项目中同类清单+同定额的高置信主材
3. `quota_material_template`
4. 从清单描述中抽取

主材结果建议统一成：

```python
material_recommendation = {
    "bill_id": "...",
    "quota_id": "C8-1-23",
    "materials": [
        {
            "name": "镀锌钢管",
            "spec": "DN150",
            "unit": "m",
            "qty_rule": 1.03,
            "price": 58.0,
            "price_source": "official_info",
            "confidence": 0.88,
            "evidence": ["quota_template", "historical_project"],
        }
    ],
    "risk_flags": [],
}
```

### 4.5 综合单价检查层

这里不要一上来做“自动组价引擎”，优先做“检查引擎”。

原因很简单：你现在手里已经有广联达和项目历史价，先把“错价、漏价、跳价”挡住，ROI 最高。

建议新增：

- `src/unit_price_checker.py`
- `src/price_feature_builder.py`
- `src/price_anomaly_store.py`

第一阶段只做四类检查：

1. `无价`
   有清单、有定额、有主材，但没有综合单价。
2. `离群价`
   同省、同类、同规格、同时间窗口下，显著高于或低于历史分布。
3. `结构异常`
   主材价变化不大，但综合单价异常波动。
4. `口径异常`
   单位、工程量、换算系数不一致导致的假异常。

建议输出：

```python
unit_price_check = {
    "bill_id": "...",
    "status": "pass | warn | fail",
    "score": 0.72,
    "flags": ["price_outlier", "missing_material_price"],
    "explanation": [
        "同省近6个月同类项目P50=132元/m，本条为198元/m",
        "主材价格波动不足以支撑该涨幅",
    ],
    "reference_range": {
        "p25": 120.0,
        "p50": 132.0,
        "p75": 145.0,
    },
}
```

## 6. 需要补的持久化表

## 7. 评测体系要分层，不能只测套定额命中率

建议把评测拆成四组指标。

### 7.1 编清单指标

- `compile_exact_rate`
- `compile_usable_rate`
- `unit_normalization_rate`
- `structure_parse_accuracy`
- `人工修改率`

其中最重要的是：

`usable_rate = 能直接进入套定额且不造成后续大偏差的比例`

因为编清单不一定要和人工一字不差，但必须能支撑后面匹配。

### 7.2 套定额指标

- `top1_accuracy`
- `top3_coverage`
- `rule_hit_rate`
- `experience_hit_rate`
- `low_confidence_ratio`

### 7.3 主材指标

- `material_attach_rate`
- `material_exact_rate`
- `material_price_coverage`
- `material_manual_fix_rate`

### 7.4 综合单价检查指标

- `price_anomaly_precision`
- `price_anomaly_recall`
- `false_alarm_rate`
- `manual_review_saved_rate`

## 8. 一条完整项目流水线应该怎么跑

建议最终把项目执行统一成下面 8 步。

### Step 1. 原始输入接入

输入文件进入适配器，抽取成 `raw_item[]`。

### Step 2. 编清单 / 标准化

`raw_item[] -> compiled_bill_item[]`

这一步的目标不是“绝对正确”，而是“结构稳定、专业正确、参数可抽取、可进入匹配”。

### Step 3. 套定额

`compiled_bill_item[] -> match_results[]`

直接复用现有匹配引擎。

### Step 4. 主材推荐

`compiled_bill_item + match_results -> material_results`

### Step 5. 综合单价检查

`compiled_bill_item + match_results + material_results + 历史价格事实 -> price_checks`

### Step 6. 前端审核

审核界面不再只给“定额结果”，而是同屏展示：

- 编清单结果
- 套定额结果
- 主材结果
- 价格异常提示

### Step 7. 回写

人工确认后分三路回写：

1. 编清单反馈库
2. 套定额经验库
3. 主材 / 价格事实库

### Step 8. 夜间评测与学习

每天夜里至少自动跑三种任务：

1. 编清单 benchmark
2. 套定额 benchmark
3. 价格异常检查回放

## 9. 未来 3 个阶段的实施顺序

### 第一阶段：先把“编清单中台”做出来

这是当前最该做的。

本阶段目标：

1. GQI 导出和非标清单都能产出标准 `compiled_bill_item`
2. 编清单结果可评测、可回放、可人工修正
3. 编清单输出稳定喂给现有套定额引擎

本阶段不要做的事：

- 不做图纸识别
- 不做复杂自动组价
- 不追求主材百分百自动

### 第二阶段：把主材层做成正式子系统

本阶段目标：

1. 主材推荐不再只是“从输入里捞”
2. 主材和定额关系可学习、可校验
3. 主材价格和项目结果产生稳定沉淀

### 第三阶段：把综合单价检查做成风险控制层

本阶段目标：

1. 先做检查，不急着做自动组价
2. 先用历史项目和主材事实把异常识别做准
3. 形成“人工只看异常项”的审核模式

## 11. 那些必须持续改进，不能一次性做完的点

### 11.1 编清单持续改进

这是长期第一优先级。

要持续积累：

- 非标表头映射
- 属性行识别
- 分部上下文继承
- 措施项 / 分部分项识别
- 名称标准化词表
- 单位标准化规则

### 11.2 编清单与套定额联调

单独看编清单未必有问题，但可能把后面带偏。

所以要长期看两个指标：

1. 编清单修改是否提升 `top1_accuracy`
2. 编清单修改是否降低 `low_confidence_ratio`

### 11.3 主材标准化

主材层最容易越积越脏。

必须长期维护：

- 材料别名
- 规格标准化
- 单位统一
- 同材不同名归并

### 11.4 价格口径一致性

后面做综合单价检查时，最大风险不是模型不聪明，而是口径不一致。

要长期约束：

- 含税 / 不含税
- 信息价 / 市场价 / 历史项目价
- 单位换算
- 时间窗口
- 地区口径

### 11.5 人工修正采集质量

闭环最后能不能变强，取决于人工修正是不是结构化采回来了。

必须保证人工修正至少能区分：

1. 编清单错
2. 套定额错
3. 主材错
4. 价格异常提示误报

否则全部混成“最后结果对了”，系统学不会。

## 12. 最后给一个明确判断

如果只看未来半年，最该投的不是：

- 图纸识别
- 全自动组价
- 更复杂的大模型链路

而是这三件事：

1. `编清单中台`
2. `主材推荐正式化`
3. `综合单价检查器`

排序上我建议：

`编清单中台 > 主材层 > 综合单价检查层`

原因是：

编清单不稳定，后面三层都会被污染；
主材层做起来后，价格层才有可靠输入；
价格层最后做检查器，收益最快，也最符合你现在的业务现实。
