# OB：速度与质量双约束主链方案
日期：2026-04-08

## 最新补充判断

再看一轮代码后，当前不能再简单说“速度问题还没解决”。

代码里其实已经有一批明确的速度优化：

- fast / standard / deep 分流
- exact direct / lightweight 开关
- hybrid session cache
- runtime cache + prewarm
- Universal KB timeout + cooldown
- 批量向量编码
- route 级 fastpath 阈值

所以现在更准确的判断是：

- 速度基础已经有了
- 当前主要问题不是继续压缩所有链路
- 当前主要问题是快路径误杀，和标准路径投入后的质量收益率不够高

## 代码层补充结论

### 1. 现在不是“没有快路径”

- `src/adaptive_strategy.py`
  已经有三档策略。
- `src/policy_engine.py`
  已经按 route 调 fastpath 阈值。
- `src/match_pipeline.py`
  已经有 `exact_exp_direct / lightweight_experience / lightweight_rule_prematch`。

所以同步主链并不是完全原始状态。

### 2. 现在最值得改的不是 validator

- `src/ambiguity_gate.py`
  是最小质量闸门。
- `src/final_validator.py`
  更像轻量兜底层。

删这两层，换不到太多速度，但会直接伤质量。

### 3. 现在最值得改的是 `adaptive_fast_return`

- `src/match_pipeline.py`
  当前 `fast` 路径经验库 miss 后会直接空返。

这里的问题不是慢，而是结束得太早。
这会造成：

- 明明还能继续搜
- 但首轮已经被提前判死
- 结果变成快了，但命中质量掉了

所以更准确的建议是：

- fast miss 后升级到 standard
- 不要直接空返

### 4. 第二个问题是标准链收益率

- `src/unified_retrieval.py`
  进入标准链后，多源检索投入已经不低。

如果最后还是落在：

- `insufficient_candidates`
- `small_score_gap`
- `low_param_score`

那问题就不是“还要不要更快”，而是“当前投入有没有换来足够质量”。

## 当前执行口径修正

原先口径偏向：

- 缩短主链
- 控制链路长度

现在要修正成：

1. 承认速度侧基础已具备
2. 不重复做低收益提速
3. 重点修快路径误杀
4. 重点提标准路径质量收益率
5. 保留最小质量闸门
6. OpenClaw 仍然放在异步复核链

## 下一步排序

1. `src/match_pipeline.py`
   改 `adaptive_fast_return`
2. `src/unified_retrieval.py`
   改标准检索的分层触发
3. `src/match_pipeline.py`
   用现有 `PerformanceMonitor` 拉阶段耗时分布
4. 依据真实耗时再决定是否还要继续做更深一轮性能优化
