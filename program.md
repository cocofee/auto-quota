# Auto-Quota Autoresearch Program

> 本文件引导 Claude Code 自动执行优化循环。
> 灵感来自 Karpathy 的 autoresearch 项目。

## 目标

提升 benchmark 命中率。每轮做一个最小改动，跑分验证，涨了保留，跌了丢弃。

## 整体流程（四阶段 + 日常实战循环）

```
【日常实战循环】（每天都在跑，持续喂数据）
  你接活 → 贾维斯全量跑（匹配+AI审核+纠正）→ 你确认/修正
  → 确认的结果：① 进经验库权威层 ② 喂给错误分析 ③ 扩充试卷
  → 每天约200条，一个月6000条真实数据

阶段零：准备考试（手动，1次性）
  ① OSS导入完成
  ② 重新生成全量试卷（所有省、放宽题数上限）
  ③ 跑分 → 保存为新基线

阶段一：数据质量循环（手动，每步跑分确认）
  ① 试卷纠错 → ② 权威层体检（错的降级）→ ③ 候选层晋升（对的升上去）
  → ④ 垃圾清理 → ⑤ 错误原因分析 → ⑥ 分析报告
  每步都要分析"为什么错"，积累错误模式指导后续优化

  ⑦ 通用知识库清洗（同步）
  → 91%的记录只在1省验证，跨省不可靠 → 用多省经验库交叉验证
  → 错的降级、跨省验证通过的加confidence

阶段二：自动循环（autoresearch，一晚上无人值守）
  每轮：分析错题 → 同义词/搜索词/排序/参数/知识库 → 跑分 → keep/discard

阶段三：自我出题（self-play，权威层足够大之后）
  系统自己编清单 → 自己套定额 → 自己批改 → 发现弱点 → 针对性强化
```

### 系统知识全景

```
定额库（149.2万条/25省）— 搜索目标池（字典）
  ↑ 搜索
通用知识库（43.7万条）— 清单模式→定额名称模式（教科书，跨省共享）
  ↑ 提供搜索提示
经验库（权威15万+候选28万）— 清单→定额编号（考试答案，省份专属）
  ↑ 直通匹配
批量匹配结果 — 大量实战记录，42%准确率的也有学习价值
```

四个库的关系：
- **经验库**是"答案"——直接告诉你这条清单用哪个定额编号
- **通用知识库**是"教科书"——告诉你这类清单应该搜什么名称的定额
- **定额库**是"字典"——所有可选的定额列表
- **批量匹配结果**是"作业"——做对的可以进经验库，做错的要分析原因

关键问题：
- 通用知识库43.7万条中91%只在1省验证过，跨省可靠性存疑
- 批量匹配遗留大量低分结果（42%等），这些是待消化的"错题集"

## 日常实战循环（每天都在跑，优先级最高）

实战数据比benchmark更有价值——benchmark是模拟考试，你每天跑的清单才是真实考试。

### 日常流程

```
你接到清单（群里的活）
  → 贾维斯全量跑：python tools/jarvis_pipeline.py "清单.xlsx" --province "XXX"
  → 输出：已审核Excel（正确/自动纠正/人工待确认）
  → 你确认或修正人工待确认的条目
  → 跑完后自动触发：
      ① jarvis_learn 存入经验库权威层
      ② 错误分析（为什么错了）
      ③ 如果该省有benchmark试卷，检查命中率变化
```

### 每天200条能带来什么

| 时间 | 累积数据量 | 效果 |
|------|-----------|------|
| 1天 | +200条 | 某个省的某类清单下次直通 |
| 1周 | +1400条 | 该省命中率明显提升 |
| 1个月 | +6000条 | 超过benchmark题量（4097），形成真题库 |
| 3个月 | +18000条 | 权威层翻倍，系统越用越准 |

### 实战结果怎么喂回系统

#### 第一步：存经验库（已有工具）

```bash
# 贾维斯全流程已经自动做了（--no-store可以关掉）
python tools/jarvis_pipeline.py "清单.xlsx" --province "XXX"

# 或者手动学习已确认的结果
python tools/jarvis_learn.py "已确认.xlsx"
```

#### 第二步：分析错题（需要新开发）

贾维斯跑完会有三类结果：
- **正确**（系统答对了）→ 不用管
- **自动纠正**（AI发现错了并改了）→ 分析：系统第一次为什么答错？
- **人工待确认**（AI不确定）→ 你确认后分析：系统为什么没信心？

```
自动纠正的条目 → 提取错误原因：
  - 搜到了但排序靠后？（排序问题）
  - 完全没搜到正确定额？（同义词/搜索词问题）
  - 搜到了但参数不匹配？（参数提取问题）

人工确认的条目 → 提取低信心原因：
  - 候选池里有答案但置信度低？（评分问题）
  - 候选池里根本没有？（搜索覆盖问题）
```

#### 第三步：扩充试卷（自动）

你确认的结果就是标准答案，可以自动变成benchmark题目：

```bash
# 从实战结果生成试卷（按省份分组）
python tools/generate_benchmark_papers.py --source jarvis_confirmed
```

这样benchmark题库每天都在增长，考试越来越接近真实场景。

### 实战 vs Benchmark 的关系

```
实战（每天200条，真实场景）
  ↓ 确认后存入经验库
  ↓ 错题分析发现系统弱点
  ↓ 扩充benchmark试卷

Benchmark（固定试卷，评测基准）
  ↓ 跑分量化改进效果
  ↓ 错题指导代码优化
  ↓ 验证改动不退化

两者互相喂数据，形成飞轮：
  实战越多 → 经验库越大 → benchmark分数越高 → 实战准确率越高 → 更多人用
```

### 与算法agent的关系

之前有个独立的"算法agent"（algo-agent skill），负责单次改进匹配算法。
现在这套方案覆盖并替代了算法agent的功能：

| | 算法agent（旧） | 日常实战+autoresearch（新） |
|---|---|---|
| 触发方式 | 你手动喊它 | 日常自动喂数据 + 晚上自动跑 |
| 数据来源 | benchmark错题 | 真实清单 + benchmark + 批量匹配 |
| 改进频率 | 一次 | 每天都在改进 |
| 反馈闭环 | 没有 | 实战→分析→改进→实战验证 |

不再需要单独调算法agent——这套飞轮本身就是一个**持续运转的算法优化引擎**。

为什么这个顺序？
- 先出全量试卷：题越多分越真实，后续优化方向不会偏
- 再改试卷错题：答案错了等于出错题，在错题上优化是浪费时间
- 再晋升数据：不改代码就能涨最多分（直通命中），晋升后很多"错题"自动消失
- 最后自动循环：在干净数据+正确试卷的基础上，精细打磨代码和配置

## 阶段一：数据质量循环（手动，试卷是参照物）

核心逻辑：**对的上去，错的下去，垃圾删掉，每步分析原因。**

```
试卷（参照物）
  ↓ 跑分对比
权威层数据 ←→ 候选层数据
  ↓ 发现问题        ↓ 发现好的
  降级/删除         晋升
  ↓                ↓
  分析为什么错      分析为什么对
  → 修同义词？      → 能泛化到其他省？
  → 修搜索逻辑？    → 补充知识库？
```

### 1.1 试卷纠错（先校准考试）

只改有**外部证据**的题，不改主观判断：
- 定额编号在库里查不到（编号本身是错的）
- 清单写"给水管DN25"，答案却是排水管定额（明显对不上）
- 清单本应匹配多个定额（管道+管卡+试压），答案只写了1个

**不改**：我觉得应该是另一个定额（主观判断）

```bash
# 改完跑分确认
python tools/run_benchmark.py --json-only --save --note "试卷纠错后基线"
```

### 1.2 权威层体检（找出错的，降级或删除）

```bash
# 体检：检查权威层数据是否有问题
python tools/experience_manager.py health --province "福建"

# 扫描脏数据
python tools/experience_manager.py clean --scan --province "福建"
```

拿试卷当参照物，对比权威层数据和试卷答案：
- **权威层答案 ≠ 试卷答案** → 分析谁对谁错
  - 权威层错 → 降级到候选层或删除，记录错误原因
  - 试卷错 → 修正试卷（回到1.1）
- **权威层有记录但匹配错了** → 分析原因：
  - 同义词缺口？→ 记下来，阶段二修
  - 参数没提取对？→ 记下来
  - 编号格式问题？→ 清理数据

```bash
# 降级问题条目
python tools/experience_manager.py health --fix --province "福建"

# 清理脏数据
python tools/experience_manager.py clean --fix --province "福建"
```

每轮清理后跑分确认：
```bash
python tools/run_benchmark.py --province 福建 --json-only
```

### 1.3 候选层晋升（找出对的，升上去）

```bash
# 查看各省候选层数据量
python tools/experience_manager.py stats

# 抽查数据质量（先看10条）
python tools/experience_manager.py promote --province "福建" --list --limit 10
```

晋升前用试卷验证：候选层数据如果和试卷答案一致，说明质量OK，可以晋升。

```bash
# 确认质量OK后批量晋升
python tools/experience_manager.py promote --province "福建" --source "oss_import" --all
```

晋升后跑分：
```bash
python tools/run_benchmark.py --province 福建 --json-only
```

**如果晋升后分数反而降了**，说明晋升的数据里有错的（和权威层冲突），需要：
1. 分析哪些题降了
2. 找到对应的晋升记录
3. 把错误的降回去

### 1.4 垃圾清理（删掉无用的）

什么算垃圾：
- 清单文本太短（≤2字）无法匹配
- 定额编号在库里不存在
- 重复记录（同省同清单同定额，多条）
- 明显错误的对应（给水管→电气定额）

```bash
# 去重
python tools/experience_manager.py clean --dedup --province "福建"

# 扫描并清理
python tools/experience_manager.py clean --scan --province "福建"
python tools/experience_manager.py clean --fix --province "福建"
```

### 1.5 错误原因分析（核心——系统要知道为什么错）

每发现一条错误数据（不管是权威层、候选层、还是试卷），都必须分类记录原因。
积累到一定量后，就知道系统哪里最弱、该往哪个方向改。

#### 错误分类表

| 错误类型 | 例子 | 根因 | 修复方向 |
|---------|------|------|---------|
| **同义词缺口** | "镀锌钢管"搜不到"热镀锌钢管" | 同义词表没覆盖 | 补同义词 |
| **参数提取错** | DN25提取成De25 | 参数提取逻辑bug | 改rule_family |
| **专业分类错** | 消防管道分到了给排水 | 品类路由不准 | 改category_routing |
| **搜索词偏差** | 搜"配电箱"，结果全是"配电柜" | 搜索词构建问题 | 改query_builder |
| **排序偏差** | 正确答案在候选池第3名 | 排序权重不对 | 改match_pipeline |
| **定额编号错** | 编号在库里不存在/写错了 | 源数据质量问题 | 清理数据 |
| **清单文本太模糊** | "详见图纸"、"同上" | 输入质量差，系统无能为力 | 标记为无法匹配 |
| **多定额遗漏** | 应该3个定额只给了1个 | 组合匹配逻辑不完善 | 长期改进 |

#### 记录方式

每个省做完1.2-1.4后，输出错误统计：

```markdown
## [省份] 错误原因分析
- 总错误: X条
  - 同义词缺口: X条（占比Y%）→ 具体缺哪些词
  - 参数提取错: X条 → 具体哪种参数
  - 专业分类错: X条 → 哪些清单分错了
  - 搜索词偏差: X条 → 搜了什么、该搜什么
  - 排序偏差: X条 → 正确答案排在第几
  - 定额编号错: X条 → 已清理
  - 清单太模糊: X条 → 无法修复，标记
  - 多定额遗漏: X条 → 记录待改
```

这些分析直接指导阶段二的自动循环：
- 同义词缺口最多 → 阶段二优先补同义词
- 排序偏差最多 → 阶段二优先调排序
- 参数提取错最多 → 阶段二优先改rule_family

### 1.6 分析报告（每个省做完后输出）

每个省处理完后，输出一份简要分析：

## [省份] 数据质量报告

### 1.7 通用知识库清洗

通用知识库43.7万条，但91%只在1个省验证过。"以为通用"但实际可能只对一个省有效。

#### 问题诊断

```bash
# 看通用知识库统计
python -c "
import sqlite3
conn = sqlite3.connect('db/common/universal_kb.db')
c = conn.cursor()
# 按验证省份数分布
c.execute('SELECT length(province_list) - length(replace(province_list, \",\", \"\")), count(*) FROM knowledge GROUP BY 1 ORDER BY 1')
for r in c.fetchall(): print(f'  逗号数{r[0]}: {r[1]}条')
"
```

#### 清洗策略

用经验库权威层（多省数据）交叉验证通用知识库：
- 知识库说"给水管道"→"管道安装螺纹连接"
- 查经验库：北京有这个对应、广东也有、浙江也有 → 真的通用，加confidence
- 查经验库：只有北京有，其他省没有或对应不同 → 不通用，降confidence

```
通用知识库记录 × 经验库多省验证
  → 多省一致：confidence +20，标记为"已验证"
  → 只有1省：保持原样，标记为"待验证"
  → 多省冲突：降级，标记具体差异
```

#### 通用知识库也能从经验库"学"

经验库权威层15万条来自多个省，可以自动提炼通用模式：

```
经验库中发现：
  北京: "镀锌钢管安装 DN25 丝接" → C10-1-10（管道安装 螺纹连接 DN25）
  广东: "镀锌钢管安装 DN25 丝接" → C10-1-8（管道安装 螺纹连接 DN25）
  浙江: "镀锌钢管安装 DN25 丝接" → 4-1-10（管道安装 螺纹连接 DN25）

虽然编号不同，但定额名称模式一样 → 提炼为通用知识：
  "镀锌钢管安装 DN* 丝接" → "管道安装 螺纹连接 DN*"
  验证省份: [北京, 广东, 浙江]  confidence: 95
```

### 1.8 批量匹配遗留问题消化

批量匹配跑出来大量低分结果（42%、31%、23%等），这些不是垃圾——是"做错的作业"。

#### 价值

- 做对的（高置信度）→ 可以进经验库候选层
- 做错的 → 分析"为什么错"，和试卷错题一样有诊断价值
- 低分省份（内蒙古等）→ 暴露系统对该省的短板

#### 处理方式

```
批量匹配结果
  ├── 高置信度(>=85%) → 直接进经验库候选层，等晋升
  ├── 中置信度(60-84%) → 算法agent抽样审核
  │     ├── 审核通过 → 进候选层
  │     └── 审核不通过 → 分析错因，加入错题集
  └── 低置信度(<60%) → 不进库，但统计错误模式
        → "为什么内蒙古只有42%？"
        → 缺同义词？缺经验数据？定额库不匹配？
```

这些分析结果同样喂给阶段二的自动循环，告诉它优先修什么。

## 阶段二：自动循环（Autoresearch）

### V2 总方向：全国安装优先，不再以补同义词为主

从当前 benchmark 结构看，安装卷的主要问题已经不只是“词没补全”，而是：

1. 对象识别错：这条清单到底是配电箱、阀门、配管、电缆头还是灯具
2. 家族路由错：候选一开始就进错了定额家族
3. 参数模板弱：DN、连接方式、安装方式、周长/回路/芯数没有稳定进入 query
4. 省份命名差异没有和“全国共性对象”拆开处理

因此阶段二从这一版开始，默认策略改为：

```text
全国安装专项优先
  ↓
先修对象路由 / 模板化
  ↓
再修参数抽取 / 参数入 query
  ↓
最后才做同义词、排序、轻提权
```

### V2 的四层结构

#### 第一层：对象层（先判这是什么安装对象）

优先建立稳定对象族，而不是直接丢给自然语言搜索：

- 配电箱 / 配电柜
- 阀门 / 法兰 / 管件
- 配管 / 导管 / 金属软管
- 电缆敷设
- 电缆头 / 终端头
- 灯具
- 开关插座 / 小电器
- 风口 / 散流器 / 防火阀
- 卫生器具

目标：优先压降 `wrong_tier`。

#### 第二层：模板层（按对象族拼半结构化 query）

示例：

- 配电箱 = 安装方式 + 周长档 + 成套/非成套 + 落地/悬挂
- 阀门 = 连接方式 + DN + 压力级 + 材质
- 配管 = 材质代号 + 明暗配 + 敷设部位 + 管径
- 电缆 = 敷设/终端 + 芯数 + 截面 + 电压等级
- 灯具 = 灯具族 + 安装方式 + 管数/功率档

安装卷优先按“参数模板题”处理，而不是“自然语言近义题”处理。

#### 第三层：省份适配层（全国共性 + 省级命名映射）

同一个安装对象，在不同省份的差异往往是：

- 子目命名不同
- 取档规则不同
- 材质词写法不同
- 安装方式写法不同

因此要把：

- 全国共性对象模板
- 省级定额命名映射

拆开处理，避免每次为了一个省的词面，把全国其他省一起带偏。

#### 第四层：验证层（按对象面板验，不只看总分）

阶段二不再只看总命中率，还要同时看对象面板：

- 配电箱命中率
- 阀门/管件命中率
- 配管命中率
- 电缆命中率
- 灯具命中率

如果某轮总分没涨，但对象面板明显改善，可以保留为下一轮排序/参数修复的前置条件。

## 阶段三：自我出题（Self-Play）

> 前提条件：阶段一完成后权威层数据足够多（>15万条，覆盖>10省）。

### 核心思路

系统学到足够多的"清单→定额"对应关系后，自己就是老师了：
1. **自己编清单** — 基于权威层数据，用AI变换清单文本（换说法、换参数、加噪音）
2. **自己套定额** — 用系统匹配
3. **自己批改** — 和权威层原始答案对比
4. **发现弱点** — 哪类变换后系统就答错了？

### 为什么要这么做

试卷再多也是有限的。自我出题可以：
- **无限生成训练数据** — 不依赖人工标注
- **针对性暴露弱点** — 故意出系统容易答错的题
- **避免同类型的错反复出现** — 发现一种错误模式，就批量生成同类题测试

### 自我出题的方法

#### 方法1：清单文本变换（最简单）

从权威层取一条记录：`"镀锌钢管安装 DN25 丝接" → C10-1-10`

AI生成变体：
- 换说法："热镀锌钢管管道安装 公称直径25mm 螺纹连接"
- 加噪音："镀锌钢管安装 DN25 丝接 详见图纸 按规范施工"
- 缩短："镀锌管 DN25"
- 换参数："镀锌钢管安装 DN32 丝接"（答案应该变）

正确答案已知（来自权威层），系统匹配后对比即可。

#### 方法2：错误模式复制

从阶段一的错误分析中提取模式，批量生成同类题：

```
发现：系统把"PPR给水管"经常匹配到"PVC排水管"
→ 生成100道PPR vs PVC的辨别题
→ 找出系统分错的根因
→ 针对性修复
```

#### 方法3：跨省迁移测试

同一条清单，在不同省份应该匹配不同的定额编号：

```
"镀锌钢管安装 DN25" →
  北京: C10-1-10
  广东: C10-1-8
  福建: 30501001
```

自动生成跨省测试集，验证系统在每个省都能选对。

### 自我出题的输出

生成的题目写入 `tests/benchmark_papers/自生成_[省份].json`，格式和正常试卷一致。
但要标记 `"source": "self_play"`，和人工标注的试卷区分开。

### 远期目标：自动编题 → 自动编清单

自动编题和自动编清单本质上是同一个能力，方向相反：
- **编题**：给定额 → 生成清单文本（用来测试系统）
- **编清单**：给图纸/需求 → 生成完整清单（用来干活赚钱）

阶段三练的就是"清单↔定额"的双向理解能力。练到足够强时，
从"出考题"翻转成"编清单"就是产品化的自然延伸。

### "这个定额为什么这么套"——匹配解释

每条匹配结果都应该能回答：**为什么选这个定额？**

```
清单：镀锌钢管安装 DN25 丝接
匹配：C10-1-10 管道安装（碳钢管、铸铁管）螺纹连接 DN25
理由：
  1. "镀锌钢管" ≈ "碳钢管"（同义词命中）
  2. "DN25" = DN25（参数精确匹配）
  3. "丝接" ≈ "螺纹连接"（同义词命中）
  4. 经验库直通命中（权威层，置信度98%）
```

如果系统能解释"为什么"，错了就能定位"哪一步出了问题"：
- 同义词没命中？→ 补同义词
- 参数提取错了？→ 改提取逻辑
- 排序排错了？→ 调权重

这个能力在阶段二的自动循环中尤其重要——AI分析错题时，有解释就能精准定位修复方向，没解释就只能瞎猜。

### 正常试卷（18份，~3431题）

| 类型 | 题数 | 测什么 |
|------|------|--------|

| 缺字段极简（5-10字） | ~10 | 最少信息量下的匹配 |

## OB补充（2026-03-12 评审吸收）

### 1. 阶段二增加热启动逻辑

阶段二不能每轮都从“重新分析错题”冷启动，否则容易反复发现同一个问题却因为本轮优先级被挤掉而持续搁置。

因此每轮结束时，必须把未处理问题写入 `current_priority_queue`，并在下一轮优先消费：

```yaml
current_priority_queue:
  active:
    - P1: 同义词缺口
  carry_over:
    - P2: query_builder 的电气品类搜索词偏差
    - P1: "PP-R管" 同义词缺口（高频）
    - P3: 福建园林树木类排序偏差
```

日志末尾要显式记录：

```markdown
- 本轮方向: P1 同义词
- 未处理的候选方向（下轮优先考虑）:
  - P2: query_builder 的电气品类搜索词偏差
  - P1: "PP-R管" 同义词缺口（高频）
```

### 2. 熔断机制补充“边际收益递减”检测

当前“连续3轮 discard 停止”属于硬熔断，但还需要一层“减速/换挡”逻辑：

- 最近5轮平均涨幅 `< 0.1%` → 不停止，但切换优先级方向
- 最近10轮平均涨幅 `< 0.05%` → 输出“当前方向已饱和”报告，等待人工指定新方向

即：
- `连续 discard` = 硬熔断
- `边际收益递减` = 换方向，不立即停机

### 3. 阶段三“自己批改”增加置信度门槛

Self-Play 中不能无门槛信任全部权威层，因为阶段一结束后权威层仍可能带少量噪声。

批改标准调整为：

- 只有 `authority && confidence >= 90%` 的记录，才能作为标准答案参与打分
- `confidence < 90%` 的权威记录只做参考，不计入命中率，也不产生训练梯度

### 4. 阶段一 1.3 的“分数下降”拆成两条诊断路径

“晋升后分数下降”不一定说明晋升数据本身错误，也可能是新增权威冲突。

因此 1.3 晋升后要先检查冲突，再判断是否为坏数据：

```bash
# 规划新增子命令
python tools/experience_manager.py health --check-conflicts --province "福建"
```

诊断拆分为：

1. **冲突路径**：同 bill_text / normalized_text 在权威层出现多答案，先消冲突再复测
2. **坏数据路径**：无冲突但分数下降，说明晋升记录本身有问题，按记录回退

### 5. 基线改为“全量 + 省级”双层结构

当前全量基线不足以支持精细 keep/discard，容易出现“一个省涨了掩盖另一个省跌了”的情况。

后续基线结构调整为：

```json
{
  "timestamp": "2026-03-12 21:54:51",
  "note": "数据质量循环后基线",
  "total": 0.362,
  "by_province": {
    "福建省市政工程预算定额(2017)": { "score": 0.233, "status": "阶段一进行中" },
    "福建省房屋建筑与装饰工程预算定额(2017)": { "score": 0.394, "status": "阶段一进行中" },
    "福建省园林绿化工程预算定额(2017)": { "score": 0.353, "status": "阶段一进行中" }
  }
}
```

keep/discard 判断优先看 `by_province`，不能只看全量平均分。
